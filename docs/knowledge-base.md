# Knowledge base: layered memory architecture

This project keeps its operational knowledge in layered, indexed files — a memory
system mapped to git-checked artifacts. Every behavior-changing commit leaves every
invalidated layer fresh; a deterministic pre-commit gate is the backstop. The file
names in this doc are EXAMPLES — replace them with your project's real artifacts.

## Memory horizons and layers

| horizon                  | role / analogy                        | typical artifact(s) (customize per project)                       | cadence                        | lifetime                     |
| ------------------------ | ------------------------------------- | ----------------------------------------------------------------- | ------------------------------ | ---------------------------- |
| **Working** (short-term) | "where am I right now" live state     | a handover / state doc + the active task file                     | overwritten as work advances   | current unit of work only    |
| **Episodic** (mid-term)  | what happened, milestone by milestone | a plan / changelog with append-only status lines                  | append-only                    | one entry per completed unit |
| **Semantic** (long-term) | stable system + domain knowledge      | `docs/*.md` deep docs + `docs/glossary.md`; map = `docs/INDEX.md` | edited only when truth changes | stable across units of work  |
| **Semantic (generated)** | mechanically derivable structure      | a generated index / API table / CSS-class index (via a build tool) | regenerated by the pre-commit hook | tracks the code exactly   |
| **Decision**             | why each choice was made              | `docs/decisions.md` (ADRs)                                        | append-only, immutable         | permanent record             |
| **Procedural**           | how the agents / devs operate         | `CLAUDE.md`, `.claude/agents/*`, `.claude/skills/*`               | edited when the pipeline moves | pipeline spec                |

## Invariant

**Every behavior-changing commit leaves every invalidated layer fresh.** A code change
that contradicts a semantic doc, breaks a decision rationale, or orphans a
working-memory entry is treated as a build bug — the commit does not land. **Failure to
update an invalidated layer is a bug**, not a style nit. The gate is deterministic; the
authoring path (below) is the living part of the system.

## Core principle: generate where deterministic, gate where hand-authored

A knowledge layer that a tool can rebuild from the code (a structural index, an API
table, a CSS/class index) should never be *gated* — it should be *regenerated*. The
pre-commit hook runs in two phases:

1. **Generate (auto-fix)** — for derivable docs, the hook re-runs the generator and
   re-stages the output, so that layer can never drift. Wire your generators into the
   GENERATE phase of `.githooks/pre-commit`. **Never list a generated doc in
   `rules.tsv`** — gating something the hook just rewrote is pointless.
2. **Gate (block)** — for hand-authored docs (deep docs, glossary, specs, ADRs), the
   `kb-detect` gate blocks the commit when a mapped code change leaves them stale, because
   only a human (or the curator) can author those.

This keeps the gate focused on what genuinely needs human judgement, and makes the
generated layer free.

## Self-maintenance mechanism: the `kb-detect` gate

When you commit a code change (a staged path matched by `.claude/kb/rules.tsv`), git's
pre-commit hook invokes `scripts/kb-detect.sh`:

1. Read `.claude/kb/rules.tsv` (extended-regex PATTERN → REQUIRED_ANY comma-separated
   layer files).
2. Compute the staged set (`git diff --cached --name-only`), then pre-filter out
   non-shipping paths — test doubles, fixtures, mocks (`__mocks__/`, `__fixtures__/`,
   `*.test.*`, `*.spec.*`). A tests-only change carries no knowledge surface, so it must
   not trip a rule on its own; keeping this in the detector keeps the rule patterns
   simple and honest.
3. For each remaining staged path, evaluate every rule whose PATTERN matches it.
4. For each match, enforce: **at least one file from REQUIRED_ANY must also be staged.**
   REQUIRED_ANY is matched by exact, whole-line path equality — so each entry must be a
   real repo-relative file path, not a glob or prefix.
5. If violated: print each violation (matched code path + invalidated layer set) and
   exit nonzero. The commit is blocked until every violation is fixed.

Escape hatch: `DOCS_OK=1` forces a clean exit for genuinely knowledge-neutral changes.
Fail-open: outside a git repo, or with no rules manifest, the gate skips rather than
blocks.

Example violation:

```
# Code changed: src/api/handlers.ts (public surface)
# Rule matched: ^src/.*\.(ts|tsx|js|jsx|svelte|vue|rs|py|go|…)$
# Required layers: docs/INDEX.md, docs/architecture.md
# Violation: neither staged — source change without a knowledge-layer update
```

## The authoring path: `kb-curator` agent (+ headless fallback)

When `kb-detect` fires a violation, the invalidated layer(s) get updated by either:

- the in-session **`kb-curator`** subagent (full-context, cheap, reliable), or
- the headless fallback **`scripts/kb-update.sh`** (opt-in via `KB_AUTOUPDATE=1`), which
  drives a small headless Claude run to write the updates and then re-gates.

Both follow the same discipline:

1. **Read the diff**: `git diff --cached` or a commit range.
2. **Derive truth from the repo**: read the actual implementation, git history, existing
   ADRs. Never invent behavior — if a fact cannot be derived from the code, report it as
   an open question rather than writing a guess.
3. **Apply minimal edits**: update the affected deep doc (module list / API table / rule
   catalog / data shape / glossary term); advance the working-memory "where we are" +
   append handoff facts; extend the plan/changelog status line when a unit of work
   completes; fix `docs/INDEX.md` rows + the ADR high-water mark. Record NEW decisions as
   immutable ADR entries only when explicitly instructed (deciding is the orchestrator's
   / human's job; recording is the agent's).
4. **Re-run `kb-detect.sh` until clean.**
5. **Stage the edits** (the agent leaves them staged with `git add`; the human or
   orchestrator owns the commit).

## Rules format (`.claude/kb/rules.tsv`)

Three tab-separated columns:

1. **PATTERN** — extended-regex (ERE) matched against each staged file path.
2. **REQUIRED_ANY** — comma-separated list of layer files; if PATTERN matches a staged
   path, AT LEAST ONE of these must also be staged. Matched by exact whole-line path
   equality, so each entry is a real repo-relative path (not a glob/prefix/regex).
3. **RATIONALE** — human reason shown by `kb-detect` on violation (and used by
   `kb-curator`).

Lines starting with `#` and blank lines are ignored. Order matters only for readability;
every matching rule is evaluated. Test doubles / fixtures / mocks are pre-filtered by the
detector before rules are evaluated, so they never trip a rule. Keep a final catch-all so
no source change ever lands fully undocumented; tune REQUIRED_ANY to the narrowest honest
layer.

Example line:

```
^src/domain/.*valid.*	docs/validation-rules.md	validation logic changed — update the rule catalog
```

## Integration with the sync checklist

`.claude/skills/sync-docs` is an agent's manual checklist run before every commit. It
mirrors the rules: task ticks → ADRs → deep docs → spec → plan/changelog → INDEX →
handover/state → memory. The rules make the gate mechanical; the checklist makes the
authoring visible and deliberate. **The checklist is the norm; the gate is the backstop.**

## Files

- **`docs/knowledge-base.md`** — this file; the spec of the memory architecture.
- **`.claude/kb/rules.tsv`** — the machine-readable path→layer mapping.
- **`scripts/kb-detect.sh`** — the deterministic gate (called by the pre-commit hook).
- **`.claude/agents/kb-curator.md`** — the in-session authoring agent.
- **`scripts/kb-update.sh`** — optional headless fallback updater; opt-in via
  `KB_AUTOUPDATE=1`.
- **`.githooks/pre-commit`** — wires the gate (and any `scripts/qa.sh`) into commits.

The gate is always enforced; the authoring path is flexible (living agent or headless
fallback).
