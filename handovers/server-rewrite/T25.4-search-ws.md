## Search — URL state, chips, regex, grouped results

### `lib/searchState.ts` — the single source of truth for "/search"'s URL

```ts
export interface SearchUrlState { q: string; mode: SearchMode; scopes: SearchScope[] }
export function parseSearchUrl(search: string): SearchUrlState
export function serializeSearchUrl(state: SearchUrlState): string   // no leading "?"
export function searchPath(state: SearchUrlState): string           // "/search" or "/search?..."
```

URL shape: `/search?q=<pattern>&mode=regex&scope=journal,recipes`. `q`
omitted → `""` (empty query — callers must not fetch, see below). `mode`
omitted → `"fixed"` (matches the backend's own default per T23.2). `scope`
omitted → `[]`, meaning "no restriction" both to the UI (no chip lit) and to
the backend (omitting the query param entirely rather than spelling out all
three). Round-trips losslessly for every representative state (see
`searchState.spec.ts`) — this is what makes the page reload/share-safe: the
full URL always reproduces the exact same search.

Both `SearchBar.svelte` (header, drives typing) and `SearchPage.svelte`
(chips/regex/results) import this module rather than either owning its own
notion of "what does the URL mean" — see "One input, not two" below for why
that matters.

Also here (the T23.2-flagged gap: `SearchHit.target` isn't SPA-route-shaped
for recipes):

```ts
export function hitRelpath(hit: SearchHit): string          // the file's home-relative relpath
export function hitCrumbSegments(hit: SearchHit): string[]  // cosmetic breadcrumb text, mockup style
export function searchHitHref(hit: SearchHit): string       // -> a router-navigable <a href>
export function groupSearchHits(hits: SearchHit[]): SearchGroup[]  // one group per file
```

`searchHitHref`: todo/journal/asset targets are already exactly this SPA's
route shapes (`/todo`, `/journal/2026-07-01`, `/assets/2026-07-01/x.txt` all
match the router's matcher table verbatim) — only the recipe target needs a
leading slash added (`search.py`'s `_map_hit` returns the bare relpath for
recipes, no leading `/`, unlike the other three types). `groupSearchHits`
relies on the backend's `sort_hits` ordering (todo, journal, asset, recipe;
lexical by name within a type) to guarantee one file's hits are contiguous —
no separate sort/group-then-sort step needed.

### `lib/searchHighlight.ts` — best-effort `<mark>` wrapping

`SearchHit` (T23.2) carries a match `col` but no match *length*, so pixel/
byte-exact reproduction of ripgrep's match span isn't attempted.
`highlightMatch(text, query, mode)` re-runs the query against `hit.text`
client-side (case-insensitively, since `rg` always runs with `-i` —
`search.py`'s `_RG_BASE_ARGS`) — literal substring in fixed mode, a real
`RegExp` in regex mode — and returns `{before, match, after}` or `null` if
nothing matches (empty query, no match, or an unparseable regex — never
throws). Purely a display nicety, not correctness-critical: if it can't find
a match, the raw line still renders unhighlighted.

### `SearchPage.svelte` (`src/lib/components/`)

Renders the toolbar (three `.chip` scope buttons + the `.switch` regex
toggle, both from `mockups/search.html`) and the grouped results
(`.search-group`/`.search-hit`/`.hit-line`/`.hit-text`/`mark.hit-match`, all
pre-existing app.css classes from T22). Reads `urlState` via
`$derived(parseSearchUrl(location.search))` — this re-derives on **every**
`router.current` reassignment, including query-only `replaceState`
navigations, because `router.navigate()` (T25.1) always builds a fresh
`Route` object even when `path` is unchanged; a component effect that reads
`router.current` (not a destructured field) reruns on any such reassignment.
Chip clicks / the regex toggle call `pushState()` → `router.navigate(...,
{replace: true})` immediately, no debounce (an explicit click should feel
instant). The actual `search()` fetch effect reacts to `urlState.q`/`.mode`/
`.scopes` — **empty `q.trim()` skips the fetch entirely** and shows a "type a
query" hint instead (an empty/missing search query is deliberately not an
error state here, unlike the backend's own 422 for a truly missing `q` param
— the SPA just never sends the request).

### `SearchBar.svelte` — one input, not two

`mockups/search.html` has no second query input in the results view — the
header search box **is** the query input, even while already on `/search`.
So `SearchBar` now has two modes:

- **Off `/search`**: unchanged stub behavior — type freely, `Enter` navigates
  to `/search?q=...` (a real history push, since arriving fresh).
- **On `/search`**: every keystroke also live-updates the URL, **debounced
  300ms**, via `router.navigate(..., {replace: true})` — replace, not push,
  so backspacing through a typo doesn't leave 20 history entries. Chip/regex
  toggles in `SearchPage` use the same replace convention. The box itself
  resyncs from the URL (via an effect comparing `urlState.q` to its own
  `value`) whenever the URL changes for a reason *other* than this box's own
  typing (a link, back/forward, a fresh submit) — never fights the user's
  keystrokes since it only overwrites when the two values actually differ.

**Deliberate URL-history choice, flagged for anyone who finds it surprising
later:** every debounced-typing and chip/regex update replaces the current
history entry rather than pushing a new one. Only the *first* navigation
into `/search` (header submit from elsewhere, or a search-result-adjacent
link) is a real push. This means the back button from `/search` goes
straight to wherever you were before searching, not through every
intermediate filter tweak — standard search-as-you-type UX (matches
GitHub/Google's own behavior), not an oversight. "Shareable/back-button-safe"
per the brief is satisfied by the URL *always* fully describing the current
search (reload/copy-paste-share reproduces it exactly), not by making every
keystroke a history frame.

## Live sync — `lib/ws.svelte.ts`

### The client

```ts
export type ConnStatus = "live" | "reconnecting" | "down";
export const ws: {
  status: ConnStatus;                          // $state, read by ConnectionDot
  connect(): void;                              // idempotent; App.svelte calls it once on mount
  subscribe(watchPath: string, handlers: {
    onDoc: (payload: DocPayload) => void;
    onDeleted: () => void;
    refetch: () => void;                        // HTTP refetch, called only on a *reconnect*
  }): { ackMtime(mtimeNs: number): void; unsubscribe(): void };
};
export function useLiveDoc(
  getWatchPath: () => string | undefined,        // MUST read a $derived value, see below
  handlers: { onDoc; onDeleted; refetch },
): { ackMtime(mtimeNs: number): void };
```

Connects to `ws(s)://<same host>/api/ws` (same-origin, mirrors the existing
`/api` fetch calls; the dev proxy already forwards `/api` with `ws: true` per
T25.1's `vite.config.ts`). `App.svelte` calls `ws.connect()` eagerly in
`onMount` (alongside `router.start()`) — **not** lazily on first doc
subscribe — so `ConnectionDot` reflects real server reachability on every
page, including `/`, `/dir/*`, `/search`, not just doc-viewing ones.
`ws.subscribe()` also calls `connect()`, so this is belt-and-suspenders, not
load-bearing.

### Reconnect: backoff + jitter, re-subscribe, refetch-on-reconnect-only

`backoffDelay(attempt, opts?)` (attempt 0-based) — exponential
`baseMs * factor^attempt` capped at `maxMs`, ± `jitterRatio` random jitter.
Defaults: `baseMs=500, factor=2, maxMs=15_000, jitterRatio=0.3` → roughly
500ms, 1s, 2s, 4s, 8s, then capped at 15s forever, each ±30% jittered. Pure
and unit-tested (`ws.spec.ts`) via an injectable `random` for determinism.

On every socket close, `#scheduleReconnect()` bumps a consecutive-failure
counter and schedules `#open()` after `backoffDelay(failures - 1)`. On
`onopen`, the client re-sends `"subscribe"` for every still-registered
`watch_path` — **and**, critically, only calls each subscriber's `refetch()`
if this is a genuine *reconnect* (`#hasConnectedOnce` was already `true`),
**never** on the very first successful connect. Rationale: the page that
just mounted already did its own initial `GET /api/journal/...`-style fetch
moments earlier; refetching again on first connect would be a redundant,
silently-wasted request. A *reconnect*, though, means the socket was down
for some stretch of real time — exactly the "missed-change window" T24's
handover calls out — so `refetch()` (the page's own existing `load()`
function, reused as-is) runs to catch anything that happened while
disconnected.

`connect()` is guarded against jumping a pending backoff wait — if a
reconnect timer is already scheduled, a fresh `subscribe()` call (e.g. a page
mount while the socket happens to be down) does **not** force an immediate
new connection attempt; it just registers the subscriber so the *next*
scheduled `#open()` picks it up.

### `mtime_ns` dedupe — pure decision, `shouldRenderIncoming`

```ts
export function shouldRenderIncoming(lastSeenMtimeNs: number | null, incomingMtimeNs: number): boolean
```

`null` (nothing seen yet) always renders; otherwise renders only if the
incoming `mtime_ns` differs from the last one this subscription
acknowledged. Each `LiveSub` (the object `subscribe()`/`useLiveDoc()`
returns) carries its own `lastMtimeNs`, updated in two places:

1. Every incoming `"doc"` push (rendered or not — a deduped push still
   records the mtime, so a genuinely-new push right after an echo isn't
   itself wrongly deduped against stale data).
2. `ackMtime(mtimeNs)` — call this from `MarkdownView`'s
   `onCheckboxSuccess(mtimeNs)` (T25.2's documented plug-in point). Every
   page (`JournalPage`/`TodoPage`/`DocPage`) now does exactly this:

   ```ts
   onCheckboxSuccess={(mtimeNs) => {
     if (doc) doc.mtime_ns = mtimeNs;   // existing local UI update, unchanged
     live.ackMtime(mtimeNs);            // NEW: tell the WS dedupe this mtime is "ours"
   }}
   ```

   Without this, the fs-watcher noticing your own checkbox-toggle write (or
   the direct broadcast T24's `PATCH /api/checkbox` already triggers) would
   look like an external change and force a full `doc = payload` re-render —
   discarding the enhanced DOM (scroll position, any open lightbox, etc.) for
   no reason. Now that push arrives, matches the already-acknowledged
   `mtime_ns`, and is silently skipped.

### Scroll-preserving re-render

`withScrollPreserved(fn)` (private) captures `window.scrollY`, runs `fn`
(which triggers `doc = payload`, i.e. Svelte replacing the `{@html}` block's
content), then restores the scroll position on the next animation frame if
it drifted. This is a light-touch heuristic, not pixel-perfect —a doc whose
rendered height genuinely changed will still shift content below the fold —
but it cancels the incidental jump `{@html}` briefly collapsing/rebuilding
the container can otherwise cause. Applied automatically to every dispatched
`"doc"` push inside `WsClient`, not something page code has to opt into.

### `useLiveDoc()` — the per-page composable, and the one contract that matters

```ts
const watchPath = $derived(doc?.watch_path);
const live = useLiveDoc(() => watchPath, { onDoc, onDeleted, refetch });
```

`getWatchPath` **must** read a `$derived` value, not the raw `$state` doc
object directly (i.e. don't pass `() => doc?.watch_path`). Svelte's `$effect`
inside `useLiveDoc` tracks whatever reactive state `getWatchPath()` touches
when it runs; reading `doc` itself as a dependency would re-run the effect —
unsubscribe + resubscribe — on **every** content update to `doc` (including
the ones this very WS client just delivered), because reassigning a `$state`
variable is always a distinct reactive event regardless of whether the value
you eventually read off it changed. A `$derived` value, by contrast, only
notifies its dependents when the *computed result* actually changes (Svelte
skips re-notification for an unchanged derived value) — so the effect only
fires on a real navigation (a different `watch_path`), which is what "follows
route changes" means here: subscribe/unsubscribe track the doc actually being
viewed, once per navigation, not once per WS message. All three call sites
(`JournalPage`, `TodoPage`, `DocPage`) follow this pattern; copy it exactly
for any future doc-viewing page.

`onDeleted` in all three pages does `doc = null; notFound = true;` — reusing
each page's existing 404 `<EmptyState>` branch rather than a separate
"deleted" UI, per T24's suggestion ("or navigating away") — showing the
generic not-found card was simpler and consistent with an already-fetched
doc's file having gone missing mid-view either way.

## Deviations from the brief/mockups, assessed

1. **`refetch()` on reconnect is skipped for the very first connect.** Not
   explicitly spelled out in T24's "what the frontend WS client must
   implement" list (which just says "on connect (and on every reconnect),
   re-subscribe... and re-fetch"), but doing it unconditionally would mean
   every page's very first mount fires a redundant duplicate `GET` moments
   after its own initial `load()` already completed. Tracked via
   `#hasConnectedOnce`; flagging in case a future reviewer expects the literal
   reading of "on connect (and on every reconnect)".
2. **No heartbeat ping.** The T24 protocol supports a client `"ping"` →
   server `"pong"`, explicitly for liveness checks, but the browser
   `WebSocket` API already fires `onclose`/`onerror` reliably on a dropped
   TCP connection in every target environment (desktop browsers, localhost
   dev) — a client-driven ping/pong heartbeat on top wasn't needed to detect
   drops here, and skipping it keeps `ws.svelte.ts` smaller. Easy to add
   later (a `setInterval` sending `{type:"ping"}`, no response handling
   needed beyond "the pong arrived" as a resubscribe-canary) if a real
   deployment ever shows silently-stale sockets.
3. **`DOWN_AFTER_ATTEMPTS = 4` (down after 5 consecutive failures) is an
   invented threshold**, not specified anywhere in the plan/T24 doc — the
   brief only names three `ConnectionDot` states without saying when
   "reconnecting" becomes "down". Chosen so a single blip (server restart
   taking a couple seconds) stays "reconnecting", but a genuinely absent
   server (~30s+ of failed attempts at the capped backoff) reports "down".
   Backoff itself never gives up — this only changes the *label*.
4. **No second `.rail-section`/`.ws-legend` sync-status legend** in
   `JournalPage`'s rail (T25.3 left this as an explicit option, not a
   requirement: "add a second `.rail-section` sibling there if S25.4 wants
   it in the rail rather than just the header dot"). The header
   `ConnectionDot` alone was judged sufficient — it's visible on every page,
   the rail legend would only be one more place saying the same three words.
5. **Highlight re-matching (`searchHighlight.ts`) is best-effort**, not a
   byte-exact reproduction of ripgrep's match span (no length in
   `SearchHit`) — documented inline and above; a mismatch here only affects
   which substring gets `<mark>`-wrapped, never which files/lines are listed
   as hits.
6. **Recipe search-hit crumbs read directory segments of the relpath**
   (`["recipes","ops"]`) while journal hits show a synthetic y/m/d split
   (`["journal","2026","07","14"]`) rather than the file's real 2-segment
   directory (`["journal","2026","07"]`) — matches `mockups/search.html`'s
   demo content exactly (`journal › 2026 › 07 › 14`) rather than a
   mechanically-uniform "just the directories" rule; the crumb span is
   cosmetic text only (`Breadcrumbs.svelte`-style trails aren't reused here,
   this is a plain `<span>`), so the inconsistency has no functional cost.
