# T23.3 S23.3 — mdrender: drop CodeHilite/Pygments from fenced code, add `guess_language`

## Responsibility

The future Svelte SPA highlights code **client-side** with Shiki, so the
server must stop baking Pygments/CodeHilite markup into rendered markdown.
`render_markdown`'s fenced code blocks now emit clean, semantic HTML
(`<pre><code class="language-x">…</code></pre>`) instead of CodeHilite's
`<div class="highlight"><pre><span>…</span>…</div>` soup. A new public
`guess_language` helper gives `DocPayload.language` (for `kind == "text"`
files) a Shiki-style language id, reusing the vim-modeline sniff that
`render_file` already had.

**Not touched:** `render_file` (raw non-markdown source files, `kind ==
"text"`'s HTML rendering path if ever used — currently unused by
`docs.py`, which only reads `.text`/`.language` for that kind) is still
Pygments-backed and stays that way until it's retired in T27. This
subtask only changes `render_markdown`'s fenced-code output and adds the
language-guessing helper.

## What changed, precisely

### `render_markdown`'s fenced-code HTML shape (new, ADR D13)

`_new_markdown()` no longer registers `CodeHiliteExtension`. With
`FencedCodeExtension` alone, python-markdown's own (undecorated) fenced
handling takes over:

- A fence with a language tag (` ```python `) renders as:
  ```html
  <pre><code class="language-python">print('hi')
  </code></pre>
  ```
  Class is exactly `language-<lang>` — `lang_prefix` is python-markdown's
  own default (`"language-"`), not something this module configures; the
  `<lang>` token is whatever string followed the opening fence, lowercase
  or not, verbatim (no normalization against `_EXT_LANG_MAP` — that map is
  only for `guess_language`, a separate concern from fenced-code
  rendering).
- A fence with **no** language tag (` ``` ` alone) renders as:
  ```html
  <pre><code>plain block
  </code></pre>
  ```
  — no `class` attribute at all.
- Content is HTML-escaped by python-markdown's own escaper: `&`, `<`, `>`,
  `"` are entity-escaped; `'` is **not** (verified directly against the
  installed `markdown==3.10.2`'s `fenced_code.FencedBlockPreprocessor.
  _escape`). No Pygments `<span>` soup, no `class="highlight"` wrapper div,
  nothing else in the markup — this is exactly what Shiki's `enhance()`
  step (mentioned in the task context) should look for: select
  `pre > code[class^="language-"]`, read the language off the class
  suffix, re-highlight the (already-correct, already-escaped) text content
  in place; a bare `pre > code` with no class is the "no known language"
  case and should just get monospace styling, no highlight pass.
- Everything else about `render_markdown` — pre-filters (redaction,
  checkbox `<input>` injection, `@tag`/`@@mention` spans, ordinal
  superscripts), every *other* extension (`def_list`, `footnotes`,
  `nl2br`, `sane_lists`, `toc`, `tables`, `attr_list`, the local
  `_MermaidExtension`/`_StrikethroughExtension`), the fresh-`Markdown`-
  per-call behavior, title extraction, TOC extraction — is **byte-
  identical** to before this subtask (ADR D13's contract). Proved by: the
  full pre-existing `test_mdrender.py` suite (redaction, checkbox, mermaid,
  strikethrough, unicode, ordinal, tags/mentions, toc, title tests) passing
  unchanged, plus the acceptance suite's journal/redaction/TOC tests
  passing unchanged.
- Mermaid fences are unaffected either way: `_MermaidPreprocessor` runs at
  preprocessor priority 30, ahead of `fenced_code_block`'s 25, so
  ` ```mermaid ` blocks are rewritten to `<div class="mermaid">` *before*
  `FencedCodeExtension` ever sees them — confirmed by
  `test_mermaid_block_not_treated_as_code_fence` (no `<code>` in output)
  and the new `test_ordinary_fenced_code_not_treated_as_mermaid` (an
  *ordinary* ```` ```python ```` fence gets the clean `language-python`
  class, not `class="mermaid"`).

### `guess_language(path: Path | str, text: str | None = None) -> str | None`

New public function in `mdrender.py`. Precedence mirrors `render_file`'s
Pygments lexer lookup exactly:

1. If `text` is given and contains a vim modeline (`vim: ft=<lang>`, same
   `_MODELINE_RE` as `render_file`), that wins. The sniffed name goes
   through the existing `LEXER_MAP` alias dict (currently just
   `{"pgsql": "sql"}`) and is lowercased — e.g. `-- vim: ft=pgsql.` ->
   `"sql"`.
2. Otherwise, `path`'s filename (case-insensitively) is checked for a
   `dockerfile` prefix (`Dockerfile`, `dockerfile`, `Dockerfile.prod`,
   … -> `"dockerfile"`), then its lowercase extension (no leading dot) is
   looked up in `_EXT_LANG_MAP`.
3. `None` if nothing matches — an explicitly fine, expected outcome (not
   an error); `docs.py`'s `DocPayload.language` field is documented as
   optional for exactly this reason.

`_modeline_language(text) -> str | None` is the extracted, shared sniff
helper — `render_file` was refactored to call it too (previously it
inlined `_MODELINE_RE.search(text)` directly), so the regex and its
semantics live in exactly one place. This is a pure refactor of
`render_file`'s internals with no behavior change — all five
`TestRenderFile` tests pass unchanged.

`_EXT_LANG_MAP` (Shiki-style **lowercase** ids, keyed by lowercase
extension without the dot) — the frontend's future `lang.ts` should mirror
this exact table by hand if either side grows:

| ext | id | | ext | id | | ext | id |
|---|---|---|---|---|---|---|---|
| `py` | `python` | | `js`/`mjs`/`cjs` | `javascript` | | `rs` | `rust` |
| `sh`/`bash`/`zsh` | `bash` | | `jsx` | `jsx` | | `go` | `go` |
| `lua` | `lua` | | `ts` | `typescript` | | `xml` | `xml` |
| `vim` | `vim` | | `tsx` | `tsx` | | `ini`/`cfg`/`conf` | `ini` |
| `md`/`markdown` | `markdown` | | `json` | `json` | | `html`/`htm` | `html` |
| `sql` | `sql` | | `yaml`/`yml` | `yaml` | | `css` | `css` |
| | | | `toml` | `toml` | | `c`/`h` | `c` |
| | | | | | | `cpp`/`cc`/`cxx`/`hpp` | `cpp` |

Plus the name-based `dockerfile` rule and the modeline `pgsql` -> `sql`
alias (reusing `LEXER_MAP`, not a separate table — its values happen to be
valid both as Pygments lexer names, for `render_file`, and as Shiki ids,
for `guess_language`; if a future alias entry breaks that overlap it needs
its own map instead of reusing `LEXER_MAP`).

### `docs.py` wiring

`build_doc_payload`'s `kind == "text"` branch now calls:

```python
language = guess_language(path, text=raw_text)
```

instead of a local `_guess_language(path)` that only used
`pygments.lexers.get_lexer_for_filename` (no modeline sniff at all). Since
`raw_text` is already read off disk for the `text`/`language` fields, this
is a strict improvement: a `.txt` file with a `vim: ft=sql` modeline now
gets `language="sql"` instead of `None` (previously only the filename
extension was consulted here — `render_file`'s modeline sniff was never
reachable from `docs.py`). `_guess_language` and its now-unused
`pygments.lexers`/`pygments.util`/`collections.abc.Sequence` imports were
deleted from `docs.py` entirely.

`test_api.py::TestDocPayloadKindDispatch::test_text_file_gets_language_hint`
(untouched — its fixture `recipes/db/schema.sql` starts with
`-- vim: ft=pgsql\n`) still asserts only `doc.language is not None`,
deliberately not pinning an exact value; with the new implementation the
value is now deterministically `"sql"` (via the modeline sniff + alias),
no longer "whatever Pygments' ambiguous SQL-lexer registration happens to
pick" as the test's own comment previously explained — that comment is
now slightly stale but the assertion it explains still holds, so per the
task's boundary (only touch assertions that actually break) it was left
alone.

## Redaction embed (S23.4)

User feedback: redacted text must be click-revealable in the SPA on
localhost. The stripping pre-filter left nothing to reveal, so the
renderer can now *embed* the original content, obscured-by-default. The
frontend contract is the T22 mockups' item 4 (`T22-mockups.md`, "Design
revisions" §4): an element with class `redacted`, obscured purely by CSS
(`.redacted`: ink block, transparent text, `user-select: none`), click or
Enter/Space toggles `.is-revealed`. **The server only ever emits the
obscured state; the frontend owns the toggle, ARIA (`role="button"`,
`tabindex="0"`, `aria-pressed`), and title flip.**

### Parameter

`render_markdown(text, *, title=None, add_toc=True, embed_redacted=False)`.

- `embed_redacted=False` (default): the legacy stripping behavior,
  byte-identical — proven by the entire pre-existing `TestRedaction`
  suite passing unchanged plus an explicit
  `test_default_is_byte_identical_to_stripped_behavior`.
- `embed_redacted=True`: redacted content stays in the output, HTML-
  escaped, wrapped in `class="redacted"` markup (below).

### Exact markup contract (the SPA implements against this)

**Inline redaction** (`... !!redacted [cause]` on a non-heading line; the
whole line is replaced, exactly as in stripping mode):

```html
<p><span class="redacted">HIDDEN_VALUE</span> --- redacted (cause: CAUSE) --- </p>
```

- `HIDDEN_VALUE` = the original line's text *before* the first
  `!!redacted`, right-stripped, HTML-escaped (`html.escape`: `&` `<` `>`
  `"` `'` all entity-escaped). May be empty (a bare `!!redacted` line).
- The visible placeholder keeps today's exact wording as a plain-text
  sibling *after* the span: ` --- redacted (cause: CAUSE) --- ` with a
  cause, ` --- redacted --- ` without. (Placeholder outside the span on
  purpose: the mockup CSS makes `.redacted`'s own text transparent, so
  the placeholder stays legible while the value is obscured.)
- Selector: `span.redacted` (inline form is always a `<span>`).

**Section redaction** (`## Heading !!redacted`):

- The heading itself renders exactly as in stripping mode:
  `<h2>…<em>…redacted…</em></h2>` (placeholder heading, original heading
  text is *not* recoverable — deliberate, matching the requirement's
  "hidden section body" wording).
- The whole hidden section body — every line after the heading until the
  next same-or-shallower heading (or EOF), including blank lines and
  nested deeper headings, joined by `\n` — is embedded, HTML-escaped, in
  **one** wrapper emitted where the hidden body stood:

```html
<div class="redacted">ESCAPED_BODY_WITH_LITERAL_NEWLINES</div>
```

- Newlines inside the div are literal; the SPA should style it with
  `white-space: pre-wrap`. Nested markdown is NOT re-rendered (a hidden
  `### heading` stays the literal text `### heading`).
- A redacted section with an empty body emits no div (nothing to reveal).
- Selector: `div.redacted` (section form is always a `<div>`) — element
  name distinguishes the two forms; both carry the single class
  `redacted` from the mockup CSS contract.

### Escaping / no-re-render guarantee (implementation shape)

Hidden values never travel through python-markdown at all: `_filter_body`
plants an alphanumeric one-shot token (per-render uuid, collision-proof)
inside the wrapper and returns a token -> HTML-escaped-value map;
`render_markdown` substitutes the tokens **after** `md.convert()`. This
is load-bearing, not an optimization: inline processors would otherwise
re-render markdown punctuation inside the value (`pass*word*123` ->
`pass<em>word</em>123` — silently dropping characters from a revealed
secret; backticks/`~~` likewise). Pinned by
`test_inline_embed_value_is_not_markdown_re_rendered`.

### Gating (`docs.py`)

Both markdown-rendering builders (`build_doc_payload`,
`build_journal_payload`) grew `allow_remote: bool | None = None` and pass
`embed_redacted=not allow_remote` into `render_markdown`:

- `None` (default) -> `_allow_remote_default()`: a build-time mirror of
  `config.Settings.allow_remote`, read from the same `AWIWI_ALLOW_REMOTE`
  env var with the same pydantic bool semantics (`TypeAdapter(bool)`;
  missing/empty -> `False`; a set-but-unparseable value fails safe to
  `True` = strip). This env fallback is what makes the requirement hold
  end-to-end: `routers/api.py` (outside this boundary) calls the builders
  without the kwarg, so the default *must* reflect the deployment's real
  setting — remote mode keeps the old stripping behavior, and redacted
  values never reach non-localhost clients.
- An explicit bool overrides the environment (tests; a future router
  forwarding `app.state.allow_remote` directly).
- Deliberately NOT gated on the per-request `is_localhost` param — that
  one is Host-header-derived (spoofable) and stays reserved for the
  unrelated secret-*file* blanking, which is unchanged.
