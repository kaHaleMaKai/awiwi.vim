# S18.2 ÔÇö rewire `M.open_link`'s image branch onto `asset.resolve_image_link`

status: done
module: `lua/awiwi/init.lua`
tests: `tests/init_spec.lua`

## What

`M.open_link`'s `elseif t == "image" then` branch (was `init.lua:559-566`) no
longer inlines its own path-building logic. It now delegates target
resolution to `require("awiwi.asset").resolve_image_link(link.target)`
(added in S18.1) and, on `nil`, reports an error and returns *without*
spawning `g:awiwi_image_opener`.

New branch body:

```lua
elseif t == "image" then
  local dest = asset.resolve_image_link(link.target)
  if not dest then
    err(('cannot open link: "%s"'):format(tostring(link.target)))
    return
  end
  local opener = vim.deepcopy(vim.g.awiwi_image_opener or { "xdg-open" })
  opener[#opener + 1] = dest
  M.deps.system(opener)
```

`asset` is the existing top-of-file local (`local asset =
require("awiwi.asset")`, line 17) ÔÇö no new require added.

## Behavior change (bug fix)

**Before:** any `image`-type link target was blindly turned into
`<awiwi_home>/assets/<dir-with-dashes-as-slashes>/<name>` and spawned via
`g:awiwi_image_opener` (default `xdg-open`) ÔÇö even for targets whose parent
directory isn't a `YYYY-MM-DD` date (e.g. `/tmp/pic.png` Ôćĺ garbage path
`<home>/assets/tmp/pic.png` still got opened).

**After:** the image branch only spawns the opener when
`asset.resolve_image_link(target)` resolves to a real, date-rooted asset
path. When it returns `nil` (relative target, `http(s)://` URL, or absolute
target whose parent dir isn't `YYYY-MM-DD`), `open_link` reports
`cannot open link: "<target>"` via the same `err(...)` helper used by the
"empty type" branch, and does **not** spawn the opener.

One-line summary for `docs/architecture.md`: *image-link opening now
validates the target resolves to a dated asset path before spawning
`g:awiwi_image_opener`; unresolvable targets error out instead of opening a
garbage path.*
