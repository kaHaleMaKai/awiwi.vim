## What

New module, ~85 lines. Optional inline-image rendering in the terminal via
snacks.nvim's `image` module, wired as a silent auto-upgrade dependency (ADR
D7 â€” same seam pattern as `lua/awiwi/picker.lua`'s telescope auto-upgrade).
With no snacks.nvim on the runtimepath, every entry point silently no-ops

## Lua surface

```lua
local img = require("awiwi.img")

img.deps = { require = require }  -- specs inject fakes here

img.resolve(file, src) -> string|nil
-- pure; delegates entirely to require("awiwi.asset").resolve_image_link(src)
-- (the `file` arg is accepted, per snacks' resolver signature, but unused â€”
-- resolve_image_link only needs the link target). nil = "not ours", caller
-- (snacks) falls back to file-relative resolution.

img.enabled() -> boolean
-- vim.g.awiwi_inline_images ~= false and ~= 0 (nil/unset = enabled/true)

img.attach(bufnr) -> boolean
-- true only if the snacks backend accepted the buffer. false (silently) if:
--   disabled, snacks.nvim not installed, snacks.image not installed, or
--   snacks.image.doc.attach(bufnr) throws (pcall-guarded).
```

## `M.attach(bufnr)` order (as specced)

1. `M.enabled()` guard â†’ false if disabled. Checked *before* touching
   `require` at all, so a disabled buffer never even probes for snacks
   (verified by spec: disabled + working fake backend still returns false).
2. `pcall(M.deps.require, "snacks")` then `pcall(M.deps.require, "snacks.image")`
   â€” any failure â†’ return false.
3. `vim.treesitter.language.register("markdown", { "awiwi", "awiwi.todo",
   "awiwi.asset", "awiwi.recipe" })` â€” unconditional, idempotent.
4. Chain (never clobber) `<snacks.image module>.config.resolve`: wraps it so
   `img.resolve(file, src)` is tried first; on `nil`, falls through to
   whatever was there before (or `nil` if nothing was there).
5. `pcall(<snacks.image module>.doc.attach, bufnr)` â€” false on throw, else
   `true`.

## API-access choice (binding decision, per the brief's "pick ONE")

**Chose: the module table returned by `M.deps.require("snacks.image")`**,
NOT `_G.Snacks`. `img.lua` reads/writes `image.config.resolve` and calls
`image.doc.attach(bufnr)` where `image = M.deps.require("snacks.image")`.
Real snacks.image (per the brief) exposes both `.config` and `.doc.attach`
directly on that module table (it's the same table `_G.Snacks.image` ends up
pointing at after `require("snacks")` runs its `_G.Snacks` side effect), so
this is equivalent in production but decouples img.lua from global state and
makes the fake trivial: specs' fake `require("snacks.image")` returns one
table, and that same table is what gets mutated/inspected â€” no separate fake
global to keep in sync.

`require("snacks")` is still called first (step 2) exactly as the brief
specifies, purely to detect "is snacks.nvim installed at all" (and to trigger
its real `_G.Snacks` side effect in production) â€” but its return value is
discarded; only `require("snacks.image")`'s return value is used for
`.config`/`.doc.attach` access.

## Resolver-chain semantics

- Guard is **per-module-table**, not a single global boolean: `chained_modules`
  is a weak-keyed (`__mode = "k"`) table mapping `image module table -> true`
  once its `config.resolve` has been wrapped. In production
  `require("snacks.image")` is a singleton, so this behaves exactly like a
  one-shot flag (repeated `M.attach` calls across many buffers only wrap the
  resolver once). In specs, each test injects a *fresh* fake `snacks.image`
  table, so each test's guard starts unset naturally â€” no shared mutable
  state leaks between specs, and no test-only reset hook is needed on the
  module.
- Chained function tries `img.resolve(file, src)` first; returns that if
  non-nil; otherwise calls the previous `config.resolve` (captured at
  wrap-time) if one existed, else returns `nil`.
- Double `M.attach` on the *same* image-module table (i.e. same buffer or a
  second buffer, same snacks session) does not re-wrap â€” confirmed by spec 8
  calling `attach` twice and checking the previous/fallback resolver fires
  exactly once per `config.resolve` invocation, not twice.

## Specs (`tests/img_spec.lua`, 11 total)

1. `img.resolve` maps `/assets/2024-03-05/pic.png` â†’ asset path (via
   `with_home` scratch-tempdir helper, mirroring `asset_spec.lua`).
2. `img.resolve` returns nil for a relative src.
3. `img.resolve` delegates to `asset.resolve_image_link` (equality check,
   not reimplementation â€” covers the "pure delegation" contract point).
4. `img.enabled` defaults true when `vim.g.awiwi_inline_images` is unset.
5. `img.enabled` false for `= false`.
6. `img.enabled` false for `= 0`.
7. `img.attach` returns false with no backend (fake `require` that throws for
   every module name).
8. `img.attach` returns false when disabled, even with a working fake
   backend (guard ordering â€” `rec.attach_calls` stays empty).
9. Happy path: `img.attach` returns true, fake `doc.attach` recorded the
   bufnr, and `vim.treesitter.language.get_lang("awiwi") == "markdown"`
   afterward.
10. Resolver chain: pre-existing `config.resolve` fake is still called for a
    relative src (chaining verified via a call-count `chained.calls`
    recorder) and our resolver wins (no fallback call) for an asset-tree src.
11. Double attach doesn't stack resolvers: two `M.attach` calls on the same
    fake image module, then one `config.resolve` call on a relative src â€”
    fallback recorder shows exactly 1 call, not 2.

Helper functions in the spec file: `with_home` (copied pattern from
`asset_spec.lua`), `with_env` (snapshot/restore `img.deps.require` +
`vim.g.awiwi_inline_images`), `fake_snacks_image(opts)` (returns a fake
image-module table + a recorder), `fake_require(image_mod)` /
`no_snacks_require`.
