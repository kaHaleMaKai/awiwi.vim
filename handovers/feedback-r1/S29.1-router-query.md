# S29.1 — router: reactive querystring/hash + hash-scroll

## Root cause

`src/lib/router.svelte.ts` — the `Route` interface (name/params/path only)
strips query and hash: `navigate()` did
`this.current = matchRoute(path.split("?")[0].split("#")[0])`. Query-only
navigations produced an equivalent `Route` (same name/params/path), so
nothing router-level exposed the querystring or hash reactively — consumers
(`SearchBar.svelte`, `SearchPage.svelte`) worked around this by reading
`location.search` directly inside a `$derived`, relying only on
`router.current` being a *fresh object reference* each navigate to re-run.

The global anchor click interceptor (still in place) `preventDefault()`s
every same-origin `<a>` click, including pure hash links (e.g. TOC anchors),
then calls `navigate()` — which never scrolled anywhere, so TOC links only
rewrote the URL bar.

## New API surface

`router.current` is now typed `RouterState` (exported), which extends
`Route` with two reactive fields:

```ts
export interface RouterState extends Route {
  search: string; // mirrors URL.search — "" when absent, else "?..."
  hash: string;   // mirrors URL.hash   — "" when absent, else "#..."
}
```

- `router.current.search` / `router.current.hash` update on **every**
  `navigate()` call and every `popstate`, including query-only and
  hash-only navigations where `name`/`params`/`path` don't change — a fresh
  `RouterState` object is always assigned, so `$derived`s depending on
  `router.current` (or its sub-fields) re-run.
- Leading punctuation is kept (`"?q=cats"`, `"#section-2"`), matching
  `URL.search`/`URL.hash` semantics — same convention the pre-existing click
  handler already used (`url.pathname + url.search + url.hash`).
- `matchRoute()` itself is untouched: still `(pathname: string) => Route`
  with the original 3-key shape (its existing spec tests assert exact
  `toEqual` shapes on that 3-key object and were not touched). `RouterState`
  is built by a new private helper, `stateFor(url: URL)`, which spreads
  `matchRoute(url.pathname)` plus `search`/`hash`.

## Hash-scroll behavior

New `scrollToHash(hash: string)` helper: if `hash` is non-empty and
`document` exists, finds `document.getElementById(hash.slice(1))` and calls
`.scrollIntoView()` on it, deferred one `requestAnimationFrame` so it runs
after the route's reactive re-render (Svelte state → DOM flush → next
paint). This mirrors the existing `requestAnimationFrame`-deferred pattern
already used in `ws.svelte.ts`'s `withScrollPreserved`.

Called from both `navigate()` and the `popstate` listener, after `current`
is reassigned. Because it's a plain rAF callback keyed only off the target
element's id (not off whether the route changed), it fires correctly for:
- a normal navigation whose URL carries a hash,
- a same-path, hash-only navigation (e.g. clicking a TOC anchor while
  already on that page) — no route/component remount needed since
  `getElementById` runs against the live DOM regardless.

No-op (doesn't throw) when there's no hash, no matching element, or no
`document` (SSR/tests without DOM).

The anchor click interceptor is unchanged structurally — it still
`preventDefault()`s and calls `this.navigate(url.pathname + url.search + url.hash)`,
which now carries the scroll behavior for free.

## Files changed

- `src/lib/router.svelte.ts` — added `RouterState` interface, `scrollToHash`
  and `stateFor` helpers; `Router.current` retyped `Route` → `RouterState`;
  `navigate()` and the `popstate` handler now build state via `stateFor(new URL(...))`
  and call `scrollToHash(url.hash)` / `scrollToHash(location.hash)`.
- `src/lib/router.spec.ts` — added two new `describe` blocks: "router.current
  — search & hash" (4 tests) and "router — hash scroll" (3 tests). Imports
  `router` (not just `matchRoute`) and `vi`/`afterEach` from vitest.

## Downstream notes (for SearchBar/SearchPage rewiring units)

- `router.current.search` / `router.current.hash` are now available as a
  direct replacement for the `location.search` workaround in
  `SearchBar.svelte`/`SearchPage.svelte` (see their comments referencing
  "query-only navigations — router.navigate() always creates a fresh Route
  object"). Not rewired here — out of this unit's boundary (only
  `router.svelte.ts`/`router.spec.ts` touched) — but downstream units can now
  do `router.current.search` instead of reading `location.search`, which
  also makes those components correctly reactive to hash changes if ever
  needed.
- No breaking change to existing consumers: `Route`'s original 3 fields are
  untouched; `RouterState` only adds fields, so `router.current.name`,
  `.params.*`, `.path` all still work exactly as before in `App.svelte`,
  `SearchBar.svelte`, `SearchPage.svelte`.
- `RouterState` is exported from `router.svelte.ts` for any downstream code
  that wants to type against it explicitly.

## Boundary respected

Only touched `src/lib/router.svelte.ts` and `src/lib/router.spec.ts`. Did
not touch `App.svelte`, `SearchBar.svelte`, `SearchPage.svelte`, or
`searchState.ts`.
