# S29.2 + S29.3 — search UX fixes (header bar + /search page)

Builds on S29.1 (`handovers/feedback-r1/S29.1-router-query.md`), which made
`router.current.search` reactive. These two units rewire the search UI onto
that API and fix all five stakeholder feedback items.

## Root causes (confirmed)

1. **SearchBar** — the URL→input resync `$effect` did `if (q !== value) value = q`
   and *read `value`*, so it re-ran on every keystroke and clobbered the
   controlled input back to `urlState.q` — which was `""` off the /search
   route (hardcoded) and frozen-at-mount on it (parsed from non-reactive
   `location.search`). Net effect: "typing shows no characters".
2. **SearchPage** — `urlState` was `$derived(parseSearchUrl(location.search))`;
   `location.search` is not reactive state, so `urlState` was frozen at mount.
   Everything downstream was stale: chip `class:is-active`/`aria-pressed`,
   the regex checkbox's `checked`, and — critically — the spread base in
   `pushState({ ...urlState, ...next })`, so each toggle serialized from the
   *mount-time* state: regex couldn't be turned off (frozen `mode` always
   lost/won the same way) and scope/mode overwrote each other (each toggle
   spread a base that didn't contain the other's latest value). The
   serialize/parse logic in `searchState.ts` was already correct — untouched.
3. **/search page had no query input at all** (feedback: "search field
   missing") — only the header bar could edit q.

## Behavior decisions

### Resync rule (both components, same pattern)

A plain (non-rune) `lastKnownQ` variable tracks the last q *this component
itself produced* — set on every `oninput` and on submit. The resync effect:

```
$effect(() => { if (urlState.q !== lastKnownQ) { value = urlState.q; lastKnownQ = urlState.q; } });
```

- It reads only `urlState.q` (reactive, via `router.current.search`) and
  `lastKnownQ` (deliberately not reactive) — **never `value`** — so
  keystrokes no longer re-trigger it. That was the clobber bug.
- External URL changes (link, back/forward, other component's navigate)
  make `urlState.q` differ from `lastKnownQ` → input resyncs.
- The component's own debounced navigate lands with `urlState.q ===
  lastKnownQ` → no-op, active typing never fought.
- Chosen over the "skip while focused" alternative: simpler, works for
  programmatic focus-retained URL changes, and needs no DOM focus queries.

### Debounce

Header bar keeps its existing design: off-route, Enter/submit navigates to
`/search?q=…`; on-route, live replaceState-as-you-type with the existing
`DEBOUNCE_MS` (300 ms). The new on-page field mirrors the on-route header
behavior exactly: 300 ms debounce → `pushState({ q })` (replaceState),
preserving scope/mode because `pushState` spreads the now-live `urlState`.

### Param preservation

- SearchBar `navigateWithQuery` already spread `urlState` when on-route; now
  that `urlState` parses `router.current.search` it actually carries current
  scope/mode (before: frozen). Off-route there are no search params to keep.
- SearchPage toggles unchanged in code (`pushState({ ...urlState, ...next })`)
  — fixed purely by `urlState` becoming reactive. scope+mode+q now coexist;
  regex toggles off; chips press/unpress live (`aria-pressed` included).
- Results-fetch `$effect` already read `urlState` — re-fetches on q/scope/mode
  changes now fall out of the reactive derivation. Verified via build/check;
  no code change needed there.

## Files changed

- `server/frontend/src/lib/components/SearchBar.svelte` — `urlState` now
  parses `router.current.search`; `lastKnownQ` resync rule; submit and
  `onInput` update `lastKnownQ`.
- `server/frontend/src/lib/components/SearchPage.svelte` — `urlState` now
  parses `router.current.search` (fixes chips/regex/merge/refetch); new
  on-page `input.search-field` bound to `qValue` with the same
  lastKnownQ-resync + 300 ms-debounced `pushState({ q })`.
- `server/frontend/src/app.css` — one line: `.search-field { max-width: 480px; }`
  (field reuses the existing `.input` class for all Noir-Deco styling).
- `searchState.ts` / `searchState.spec.ts` — **not changed**: no pure helper
  was added or modified (the URL merge logic was already correct; all fixes
  are component reactivity wiring, gated by check/test/build per the unit
  brief). No new red-first tests were owed under the "new/changed pure
  helpers" rule.

Not touched: `router.svelte.ts`, `router.spec.ts` (per boundary).
