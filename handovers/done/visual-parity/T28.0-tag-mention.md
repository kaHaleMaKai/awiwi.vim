# T28.0 — mdrender tag/mention fix + class contract

Zero-context handover: read this before touching `awiwi.mdrender`'s inline-markup code or
`frontend/src/app.css`'s `.awiwi-tag`/`.awiwi-mention` rules again.

## What was broken

1. **Mention bug**: `_PERSON_TAG_PATTERN = re.compile(r"@(@[^\s,;.)}\]])")` (server.old-ported)
   only captured `@` + **one** character after the leading `@`. `@@lars` rendered as
   `<span class="awiwi-mention">@l</span>ars` — visibly broken (only `@l` colored/styled, `ars`
   plain text right after).
2. **No `#tag` support at all.** `server.old`/the port never emitted anything for `#word` tokens —
   the SPA mockups (`mockups/journal.html`, `todo.html`) show `#project-awiwi` /
   `#recipes/sourdough-starter` styled via `.tag`, but the server produced plain, unstyled text.
3. **Class-name mismatch.** `server/frontend/src/app.css` (and `mockups/tokens.css`, the T22
   ground-truth design reference) styled `.tag`/`.mention` — but the server's actual output uses an
   `awiwi-` prefix convention for everything else it emits (`awiwi-bug`, `awiwi-checkbox`, etc.), so
   even a correctly-emitted `#tag`/`@@mention` span would never have picked up the mockup's colors.

## Chosen class contract

**Server-side `awiwi-` prefix wins.** The server emits (and the frontend now styles):

| token | emitted class |
|---|---|
| `@bug` / `@change` / `@incident` / `@issue` | `awiwi-bug` / `awiwi-change` / `awiwi-incident` / `awiwi-issue` (unchanged, pre-existing) |
| `@@word` (mention) | `awiwi-mention` |
| `#word` or `#path/style-tag` | `awiwi-tag` |

Example emitted HTML (from `render_markdown`, after python-markdown conversion):

```html
<p>cc <span class="awiwi-mention">@@lars</span> about the rollout plan for
<span class="awiwi-tag">#project-awiwi</span>.</p>

<p>See <span class="awiwi-tag">#recipes/sourdough-starter</span> for the method.</p>
```

`server/frontend/src/app.css` (~line 481) was renamed selector-only, visual rule bodies unchanged
(copied verbatim from `mockups/tokens.css`, which is untouched — it's the frozen T22 design
reference, not shipped code):

```css
.awiwi-tag { color: var(--accent-brass-strong); font-weight: 600; }
.awiwi-mention { color: var(--accent-cyan); font-weight: 600; }
```

No other file in `server/frontend/src/` or its test suite referenced `.tag`/`.mention` or
`awiwi-tag`/`awiwi-mention` (checked via grep across `src/`, `tests/`), so no other frontend changes
were needed.

## Regex semantics (`server/src/awiwi/mdrender.py`)

All three tag-ish substitutions now go through **one combined regex**, `_INLINE_MARKUP_RE`, applied
once per line inside `_filter_body` (replacing the old two separate `.sub()` calls):

```python
_TAG_PATTERN = re.compile(r"@(?P<type>bug|change|incident|issue)\b")   # unchanged
_PERSON_TAG_PATTERN = re.compile(r"@@[\w-]+")                          # fixed: whole token
_HASHTAG_PATTERN = re.compile(r"(?<!\()(?<![\w#])#[\w][\w/-]*")        # new
_CODE_SPAN_RE = re.compile(r"`[^`\n]*`")                               # new (pass-through)
_INLINE_MARKUP_RE = re.compile("|".join([...four patterns above, named groups...]))
```

`_INLINE_MARKUP_RE.sub(_wrap_inline_markup, line)` tries alternatives in order at each match
position (normal regex alternation semantics): a code span always wins if present at that position,
so its content is returned untouched; otherwise whichever tag/mention/hashtag pattern matched gets
wrapped in its span.

**Mention (`@@word`)**: terminates at the first character outside `[\w-]` (any whitespace or other
punctuation). Trailing punctuation (`@@lars.` / `@@lars,`) is correctly excluded from the token now
— previously only 1 char was captured so this didn't matter, now the whole run of word chars/hyphens
is captured and nothing more.

**Hashtag (`#word` / `#path/style`)**: `[\w][\w/-]*` — first character after `#` must be a word
char (this is also what excludes headings, see below), then any run of word chars, hyphens, or
slashes (slashes so `#recipes/sourdough-starter` is one token, not two). Terminates at the first
char outside that class (`.`, `,`, whitespace, etc.).

Two lookbehinds gate the `#`:
- `(?<![\w#])` — previous char isn't a word char or another `#`. Excludes `word#tag` (mid-word,
  unlikely but cheap to exclude) and `##foo` (a doubled/malformed `#`, previous char is `#`).
- `(?<!\()` — previous char isn't `(`. Excludes markdown link hrefs written as
  `[text](#anchor)` — the `#anchor` right after the opening paren is a URL fragment, not a tag.

## Edge cases explicitly excluded (and why no extra code was needed)

- **Headings** (`# Title`, `## Section`): a heading marker is *always* `#`(+) followed by a space
  (`_HEADING_RE = re.compile(r"^(?P<marker>##+) ")` elsewhere in this file enforces that same
  invariant for the redaction-hiding logic). Since `_HASHTAG_PATTERN` requires an **immediate** word
  character after `#` (no space allowed), it structurally cannot match a heading marker — no
  separate "is this a heading line" check was needed. A heading whose *title text* happens to
  contain a real hashtag later in the line (`# Notes about #project-awiwi`) still gets that later
  occurrence wrapped correctly, since only the leading `# ` fails to match, not the rest of the line.
- **Markdown link hrefs** (`[text](#anchor)`): excluded via the `(?<!\()` lookbehind, see above.
  TOC-generated anchors (`href="#section-one"`) are never at risk at all — those are emitted by
  `TocExtension` *after* `_filter_body` already ran, so they never pass through this regex.
- **Inline code spans** (`` `#project-awiwi` ``, `` `@@lars` ``): matched by `_CODE_SPAN_RE` as the
  first alternative in `_INLINE_MARKUP_RE`, so a code span's content is always returned byte-for-byte
  unchanged (regardless of what's inside), even though it's on the same line as other real tags.
- **Fenced code blocks** (```` ``` ````, including ```` ```mermaid ```` diagram source):
  `_filter_body`'s per-line loop now tracks an `in_fence` boolean, toggled by `_FENCE_DELIM_RE`
  (`^```` , no leading-whitespace tolerance — same convention as the existing
  `_MermaidPreprocessor._FENCE` check). While `in_fence` is `True`, lines (including the delimiter
  lines themselves) are appended to the output completely unprocessed — no tag/mention/hashtag
  substitution, no ordinal-superscript substitution either. This is a **pre-existing gap that was
  live/shipped before T28.0** (`@bug` inside a fenced code block used to get wrapped in a span,
  which then got HTML-escaped by `FencedCodeExtension` into visible `&lt;span...&gt;` garbage in the
  rendered code — reproduced and confirmed during this work). Fixed as part of the same root-cause
  pass since the task asked for code-region avoidance and the fix is a natural extension of the same
  per-line loop.
- Fence tracking is skipped entirely while a redacted section is actively hiding content
  (`_filter_body`'s `hide` state) — hidden lines never run through *any* pre-filter regardless
  (pre-existing invariant, see `_flush_hidden`'s docstring), so fence state simply doesn't change
  for lines that are already invisible; no interaction to worry about.
