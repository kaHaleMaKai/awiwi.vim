# T28.2 — Screenshot harness (`shoot.mjs` + README)

Zero-context handover: read this before rerunning the harness, changing `pairs.json`, or
writing T28.3's haiku comparator prompts.

## What this is

`server/tests/visual/shoot.mjs`: one dep-free node script that shoots every pair in
`pairs.json` (mockup `file://` page + SPA route) via raw Chrome DevTools Protocol (no
puppeteer/playwright), then post-processes each pair with ImageMagick — pad to common
canvas, RMSE diff, 1440x760 band slices — plus `metrics.json`. `server/tests/visual/README.md`
covers invocation/flags/output-layout/how-to-add-a-pair in full; this handover covers the
iter0 run, the sandbox finding, and two real bugs the run itself surfaced.

## Exact invocation

```sh
cd server/tests/visual
node --experimental-websocket shoot.mjs --out <dir>
```

(`--experimental-websocket` is load-bearing on node 20.19.2 — global `WebSocket` throws
`ReferenceError` without it; verified directly, not just per the plan.)

Ran for real at:

```sh
node --experimental-websocket shoot.mjs --out /tmp/.../scratchpad/shots-iter0
```

## Sandbox finding

**Sandboxed chromium launch worked on the first try** — `--no-sandbox` was never needed on
this machine. The harness still implements the fallback (tries sandboxed first, retries once
with `--no-sandbox` if the process exits early or prints no `DevTools listening on ws://...`
line within 10s) since the plan calls it out as an open question and CI/other machines may
differ; `usedNoSandbox` is logged either way (`shoot.mjs: chromium up on devtools port NNNNN
(sandboxed launch worked, --no-sandbox not needed)`).

## Two real bugs found and fixed during iter0 verification

The task asked for a spot-check of one page's bands; that surfaced a genuine harness bug, so
the check was widened to every "dark" page before calling iter0 final.

### 1. Theme state leak across SPA targets (harness bug, fixed)

First iter0 run: `journal`/`journal-light` shot correctly, but **`dir-root`, `404`, and by
extension every "dark" SPA page shot *after* a light-theme pair rendered in the light theme**
even though `pairs.json` says `"theme": "dark"` for all of them. Root cause: the original
script only ran `localStorage.setItem('awiwi.theme','light')` for `theme: "light"` pairs and
did nothing for dark ones. All SPA targets share one browser profile (one origin →
one `localStorage`), so once `journal-light`'s injected script set the key, it silently
persisted into every subsequent target's `Page.addScriptToEvaluateOnNewDocument`-injected
document (a *new tab*, not a *new origin* — `localStorage` isn't per-tab). Caught by looking
at `dir-root-spa-band-00.png` and seeing a cream/light background with `LIGHT` highlighted in
the theme toggle instead of the expected dark chrome.

**Fix** (`shoot.mjs`, `themeScriptSource`): every SPA shot now *always* explicitly asserts
`localStorage.setItem('awiwi.theme', 'light'|'dark')` matching its own pair's `theme` field,
not just the light ones. Confirmed fixed by re-running and re-viewing `dir-root-spa-band-00`
and `recipe-audit-spa-band-00` — both now show the correct dark chrome (`DARK` highlighted).

This inflated RMSE across the board in the first run (e.g. `dir-root` read 0.866 normalized
RMSE — light-vs-dark, not a real style diff) — ignore any earlier numbers; the table below is
post-fix.

### 2. `recipe-audit`'s `spaUrl` needed `.md` (pairs.json bug, fixed)

`pairs.json` had `"spaUrl": "/recipes/sourdough-starter"` for the unpaired `recipe-audit`
entry. The SPA's `Recipes.svelte` route builds `DocPage path={`recipes/${rest}`}` verbatim
from the URL segment and calls `GET /api/doc/recipes/${rest}` with no extension massaging —
confirmed by curl: `/api/doc/recipes/sourdough-starter.md` → 200, `/api/doc/recipes/sourdough-starter`
(no extension) → 404. So the URL without `.md` silently rendered the SPA's real 404 page
(`recipes/sourdough-starter doesn't exist, or was moved.`) — a **valid page load**, not a
harness error, which is why nothing failed loudly; only a screenshot spot-check caught it.

**Fix**: `pairs.json`'s `recipe-audit.spaUrl` → `/recipes/sourdough-starter.md`. Re-shot;
`recipe-audit-spa-band-00.png` now shows the actual recipe content (table, steps, `#tag`)
instead of the 404 glyph. Breadcrumb reads `sourdough-starter.md` (extension visible in the
crumb) — noted as a real content quirk for T28.3/escalation, not a harness bug: worth deciding
whether the crumb should strip `.md` for recipes the way journal dates don't show one, but
that's styling-parity/behavior scope, out of bounds for this harness task.

**Lesson embedded in the README**: a wrong `spaUrl` or a theme-leak-style bug still produces
a plausible-looking PNG and RMSE number, not a script error — always spot-check at least the
band-00 PNGs of new/changed pairs before trusting the metrics.

## Capture quirks

- **Every non-journal page capped at exactly 1000px height on both sides** (mockup and SPA
  alike) — not a bug: these pages' content is genuinely shorter than the 1000px viewport, and
  the layout's `min-height: 100vh`-style baseline means `captureBeyondViewport` reports
  exactly the viewport height rather than a shorter "true" content height. Confirmed the
  mechanism itself works correctly — the `journal`/`journal-light` pairs *do* capture beyond
  the viewport (1777px, content genuinely overflows) — so this is expected fixture-content
  behavior, not a harness ceiling.
- **No page hit the 15s stability cap** in this run. The plan flagged mermaid/Shiki settling
  as a risk; in practice both settle well inside 2s for this fixture's content (small mermaid
  diagram, short code fences). If a future fixture page has heavier content, expect
  `stable after CAP(15000ms)` to start appearing in the log — that's handled gracefully (last
  capture used, script continues), just slower.
- Mockup pages never needed padding relative to their SPA counterpart in this run (all
  `mockupDims === spaDims` per pair) — the one prior run where `dir-root`/`404`/etc. looked
  padded-different was purely the theme-leak bug (light vs dark backgrounds happened to
  render the same viewport height here, so it showed up as RMSE, not as a dimension mismatch —
  worth remembering for T28.3 that dimension equality alone doesn't rule out a theme mismatch).

## Spot-check impression (journal mockup vs SPA, band 0)

Structurally very close — same header/breadcrumb/search-bar/ws-dot/theme-toggle chrome, same
"DAILY JOURNAL" eyebrow + big date heading + prev/next nav cards + TOC rail with the same six
section names, same redacted-pill treatment in Work. Visible real gaps for T28.3 to chase:
`#tag`/`@@mention` render as plain unstyled text in the SPA (mockup shows them in gold/olive —
this is the known T28.0-deferred backend fix, not new), the date heading format differs
("Tuesday, July 14 2026" mockup vs "Tue, 14th July 2026" SPA), breadcrumb segment style differs
(`2026 › 07 › 14` vs `2026 › 07 › 2026-07-14`), and the TOC rail renders as a bulleted list
(SPA) vs a plain heading list with an active-item accent bar (mockup). None of these are
harness artifacts — same content, same viewport, same theme, real style/behavior deltas.
