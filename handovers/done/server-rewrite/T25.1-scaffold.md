# server-rewrite / T25.1 — frontend SPA scaffold

## Responsibility

Scaffold `server/frontend/` (Svelte 5 + TypeScript + Vite 7 SPA) that will
replace the Jinja2 templates as the awiwi note viewer: vite-svelte-ts
project layout, the Noir-Deco design tokens ported into `app.css` with
self-hosted fonts, a hand-rolled runes router, theme state, the typed
`/api` fetcher surface, and an `App.svelte` shell with the shared header.
Route *views* are placeholder stubs — S25.3 fills them in. This unit does
not touch the backend, docs, or anything outside `server/frontend/` +
`.gitattributes`/`.gitignore`.

**Boundary:** `server/frontend/**` (new), root `.gitattributes` (new), this
handover, `.claude/progress/engineer-s25.1.md`. Nothing else touched —
`server/src`, `server/tests`, `docs/`, `mockups/` all byte-for-byte as
before (mockups/tokens.css and mockups/fonts/ were only *read*, then
copied — the originals under `mockups/` are untouched).

## Conventions for S25.2–S25.4

- **Components** (reusable, no route semantics) live in
  `src/lib/components/`. **Route views** (one per router route name) live
  in `src/routes/`, one file each, named to match the router's `RouteName`
  (`Home`, `Dir`, `Todo`, `Journal`, `Asset`, `Recipes`, `Search`,
  `NotFound`) — App.svelte's `{#if router.current.name === ...}` chain is
  the only place that imports them, so adding a new route view is a
  two-line change there (new import + new branch), never a router change.
- **`enhance/` pipeline** (S25.2: Shiki, mermaid, checkbox wiring, media
  rewrite, lightbox) should live in `src/lib/enhance/` — not created yet,
  intentionally, since S25.1 has no markdown-HTML DOM to enhance.
- Everything reactive that needs runes outside a `.svelte` file uses the
  `*.svelte.ts` extension (`theme.svelte.ts`, `router.svelte.ts`) — that's
  what makes the Svelte compiler process `$state`/`$derived` in those
  files. Plain `.ts` files (`api.ts`) have no runes and are ordinary
  TypeScript.
- Tests are colocated (`router.spec.ts` next to `router.svelte.ts`), not in
  a separate `tests/` tree — standard Vitest convention, keep it for
  S25.2+.
- No Shiki, no mermaid, no drawio vendor script yet — those are S25.2/S25.3
  additions per the plan; don't add them early.

## Theme API (`src/lib/theme.svelte.ts`)

```ts
export type Theme = "dark" | "light";
export const theme: {
  current: Theme;           // reactive ($state) — read directly in templates
  set(next: Theme): void;   // persists + flips <html data-theme> + stamps .theme-transition for 350ms
  toggle(): void;
};
```

- Persistence key is **`localStorage['awiwi.theme']`** (`'dark' | 'light'`,
  default `'dark'`) — per this subtask's brief, literally. **Note the
  mismatch with the T22 mockups**, whose `mockup.js` demo script uses
  `awiwi-theme` (hyphen). The mockup script is throwaway/non-contractual
  (see T22 handover: "static mockups... discardable"); `theme.svelte.ts`
  is the real contract going forward. Flagging so nobody "fixes" this key
  to match the mockup later without realizing it was a deliberate literal
  instruction, not a typo.
- **Zero-flash on load**: `index.html` has a small blocking (non-module)
  inline `<script>` in `<head>` that reads `localStorage['awiwi.theme']`
  and sets `data-theme` on `<html>` *before* `main.ts`/Svelte ever runs.
  `theme.svelte.ts`'s initial `$state` value is seeded by reading that
  same `data-theme` attribute back off `document.documentElement` (falling
  back to the stored value, then `'dark'`) — so the store's initial value
  and the already-painted DOM never disagree.
- `ThemeToggle.svelte` is fully wired (not a stub) — two buttons, `.is-active`
  class bound to `theme.current`, matching the mockup markup/CSS exactly.

## Router API (`src/lib/router.svelte.ts`)

```ts
export type RouteName = "home" | "dir" | "todo" | "journal" | "asset" | "recipes" | "search" | "notfound";
export interface Route { name: RouteName; params: Record<string, string>; path: string; }

export function matchRoute(pathname: string): Route;   // pure, unit-tested directly

export const router: {
  current: Route;                                        // reactive ($state)
  navigate(path: string, opts?: { replace?: boolean }): void;
  start(): void;                                          // wires popstate + global <a> click interception; idempotent
};
```

Matcher table (order doesn't matter except the catch-all is last):

| pattern | name | params |
|---|---|---|
| `/` | `home` | `{}` |
| `/dir/*` | `dir` | `{ rest }` (may be `""`) |
| `/todo` | `todo` | `{}` |
| `/journal/:date` | `journal` | `{ date }` |
| `/assets/:date/:file` | `asset` | `{ date, file }` |
| `/recipes/*` | `recipes` | `{ rest }` (may be `""`) |
| `/search` | `search` | `{}` |
| `/*` | `notfound` | `{}` |

- `App.svelte` calls `router.start()` once in `onMount` — this attaches the
  `popstate` listener and the document-level click listener that
  intercepts same-origin, unmodified, non-`_blank`, non-`download` `<a>`
  clicks and turns them into `history.pushState` + a `matchRoute` re-run
  (no full page reload). External links, modified clicks (ctrl/cmd/shift/
  alt-click), `target="_blank"`, and `download` links are left alone.
- `router.current` is `$state`, so any component that reads
  `router.current.name`/`.params` directly (not destructured into a plain
  variable first) re-renders on navigation — see `App.svelte`'s route
  switch for the pattern to copy.
- `matchRoute` is exported specifically so it can be tested without
  touching `window`/`document` — see `router.spec.ts` (12 cases: one per
  route, two "doesn't match with a missing required segment" negative
  cases, one trailing-slash case).

## `/api` fetcher surface (`src/lib/api.ts`)

Mirrors `handovers/server-rewrite/T23.2-api-routes.md` exactly — **that
document, not `server/src`, is this file's source of truth**; if the
backend contract changes, update both together.

```ts
getMeta(): Promise<MetaPayload>
getJournal(dateStr: string): Promise<DocPayload>
getTodo(): Promise<DocPayload>
getDoc(path: string): Promise<DocPayload>
getDir(path?: string): Promise<DirPayload>
rawUrl(path: string, opts?: { download?: boolean }): string   // URL builder, not a fetcher — bind to <img src>/<a href>
patchCheckbox(body: CheckboxPatchRequest): Promise<CheckboxPatchResponse>
search(params: { q: string; mode?: "fixed"|"regex"; scope?: ("journal"|"assets"|"recipes")[] }): Promise<SearchHit[]>
```

Types exported: `DocKind`, `DocType`, `BreadcrumbPayload`, `NavPayload`,
`DocPayload`, `DirEntry`, `DirPayload`, `SearchHit`, `MetaPayload`,
`CheckboxPatchRequest`, `CheckboxPatchResponse`, `SearchMode`,
`SearchScope`, `ApiError` (thrown for any non-2xx; `.status` + `.detail`,
where `.detail` is the JSON body's `detail` field — a string for most
routes, FastAPI's validation-error array for `/api/search` 422s).

Notes for S25.2/S25.3:
- `mtime_ns` is typed `number` but is a Python nanosecond epoch int wider
  than JS's safe-integer range — fine for equality/round-tripping (as the
  checkbox PATCH protocol requires), **do not** do arithmetic on it.
- `SearchHit.target` is a legacy template-route-style path (e.g.
  `/journal/2026-07-01`), **not** a `watch_path` relpath and **not** an
  `/api/doc/...` URL — per T23.2's own flag, search-result click-through
  needs its own target→relpath mapping if the SPA wants to route hits
  through `getDoc`. Not built here — out of this subtask's scope.
- All path-taking fetchers percent-encode each path *segment*
  individually but preserve literal `/` (the backend's `{path:path}`
  converters accept raw slashes) — use `getDoc`/`getDir`/`rawUrl` rather
  than hand-building URLs so this stays consistent.

## App shell (`src/App.svelte`)

Header markup follows the mockups' `.app-header` structure (brand link,
`Breadcrumbs`, spacer, `SearchBar`, `ConnectionDot`, `ThemeToggle`) — same
DOM shape as `mockups/journal.html`'s header, so `app.css`'s
`.app-header`/`.breadcrumbs`/`.theme-toggle`/`.ws-indicator` rules apply
unmodified. `Breadcrumbs` currently gets a single hardcoded `{ name:
"awiwi", target: "/" }` crumb — **this is a placeholder**; each route view
fetches its own `DocPayload`/`DirPayload` (which carries a real
`breadcrumbs` field) in S25.3, and should replace this with the fetched
trail (e.g. via a small shared `$state`/prop threaded up, or lifting the
crumb array into the router — implementation choice left to S25.3, not
decided here to avoid a premature abstraction).

Main content area is a route switch on `router.current.name`, rendering
one of the eight stub views in `src/routes/`, passing `router.current.params`
fields through as typed props (`date`, `file`, `rest`).

## Deviations from the brief / plan, assessed

1. **Vite/plugin versions pinned below latest.** `npm create vite@latest`
   (as of this run) scaffolds `vite@^8.1.1` + `@sveltejs/vite-plugin-svelte@^7.1.2`,
   but the plan explicitly pins `vite@^7`. Downgraded to `vite@^7.3.5` +
   `@sveltejs/vite-plugin-svelte@^6.2.4` (the last vite-plugin-svelte line
   whose peer range includes vite 7, per `npm view ... peerDependencies`).
   `svelte@^5`, `svelte-check@^4`, `typescript@~6.0.2` (as scaffolded),
   `vitest@^4.1.10` (peer-compatible with vite 7), `happy-dom@^20.10.6` —
   no deps beyond what the plan/brief names.
2. **Theme localStorage key is `awiwi.theme` (dot), not `awiwi-theme`
   (hyphen) used by the mockups' demo `mockup.js`.** This subtask's brief
   states the dotted key literally; documented as a deliberate,
   flagged divergence above (Theme API section) rather than silently
   "fixed" to match the mockup.
3. **`ConnectionDot` defaults to `"down"`**, not `"live"` — there's no WS
   client wired up yet (that's S25.4, depends on T24), so "down" is the
   honest default rather than a fake "live" indicator with nothing behind
   it.
4. **Breadcrumbs in `App.svelte` are a single hardcoded placeholder
   crumb**, not real per-route data — noted above, left for S25.3 since
   real breadcrumbs require a fetched `DocPayload`/`DirPayload`, which no
   route view fetches yet in this subtask.
5. Kept the default Vite `public/favicon.svg` rather than pulling
   something from `mockups/` (mockups have no favicon asset) — cosmetic,
   flag if design wants a real brand mark here.
