# T23.2 S23.2 — `/api/*` JSON routes (the frozen SPA API contract)

## Responsibility

Wires the S23.1 payload builders (`awiwi.docs`) and schemas (`awiwi.schemas`)
into real HTTP routes: `server/src/awiwi/routers/api.py`, all under the
`/api` prefix. This is **the** document a frontend engineer building the
future Svelte SPA needs to read — it is written to be usable with zero other
context. Additive: the legacy template routes (`routers/pages.py`,
`routers/assets.py`, `routers/actions.py`) are byte-for-byte untouched; their
own test suites still pass unmodified.

## Boundary (touched)

- `server/src/awiwi/routers/api.py` (new) — every endpoint below.
- `server/src/awiwi/search.py` — extended `build_rg_args` with keyword-only
  `fixed`/`scopes` params (see below). `parse_search_output`/`sort_hits`
  untouched.
- `server/src/awiwi/app.py` — one import + one `include_router` line,
  registered `assets -> actions -> api -> pages` (see Route-ordering below).
- `server/tests/test_api.py` — extended with 52 new HTTP-level tests (see
  Tests).
- `server/tests/test_search.py` — extended with 7 new tests for the
  `build_rg_args` additions.

**Not touched:** `schemas.py`, `docs.py`, `httputil.py`, `content.py`,
`checkbox.py`, `mdrender.py`, `templating.py`, `config.py`, any existing
router or template, `conftest.py`. Verified via `git status` (only the files
listed above show as modified/new).

## Route-ordering — why `api` must sit before `pages`

`routers/pages.py` registers `GET /{path:path}` as an app-wide catch-all
(serves any file under `home`), which necessarily also matches any
`/api/whatever` path string. If `pages.router` were included before
`api.router`, an unmatched `/api/typo` request (or any `/api/*` path this
router doesn't recognize) would fall through to that catch-all and either
404 as an **HTML** page or — worse — actually resolve and serve some file
whose home-relative path happens to start with `api/`.

Fixed two ways, both required:

1. `app.py` includes routers in this order: `assets -> actions -> api ->
   pages`. Starlette matches routes in registration order, so `api`'s own
   routes (including its catch-all, next point) are checked before `pages`'
   catch-all ever gets a chance.
2. `api.router` ends with its own catch-all, `GET /api/{rest:path}`,
   registered **last** among the `api` router's routes (Starlette also
   matches routes within one router in registration order). Any `GET
   /api/<unmatched-path>` hits this and returns a clean JSON 404 —
   `{"detail": "no such API route: /api/<path>"}` — instead of ever reaching
   `pages`' catch-all.

Caveat: the catch-all above is `GET`-only. A non-GET request to an
unmatched `/api/*` path (e.g. `PUT /api/bogus`) isn't covered by it and — in
the current routing table — wouldn't be covered by `pages`' catch-all either
(also `GET`-only), so it 404s via Starlette's own "no route matched"
handling rather than this router's JSON handler. Not exercised by tests;
flagged for whoever tightens this later if it matters.

## Auth / localhost posture

Identical to every other router: the app-wide `localhost_only` middleware
(`app.py`) 403s any non-localhost request unless `AWIWI_ALLOW_REMOTE=1`
(`Settings.allow_remote`). `/api/*` routes get no separate gate for this —
they run after the same middleware as `pages`/`assets`/`actions`. Two of the
routes (`/api/doc/...`, `/api/raw/...`) additionally consult
`httputil.is_localhost(request)` themselves for the **secret-file** content
gate, independent of (and in addition to) that middleware — see each
route's section.

## Route list

All paths below are relative to `/api`. All responses are JSON except
`GET /api/raw/{path}` (raw bytes). Errors use FastAPI's standard
`HTTPException` shape: `{"detail": "<message>"}`.

### `GET /api/journal/{date_str}`

Journal-day page as `DocPayload` (see schema below). `date_str` accepts
everything the legacy `/journal/{date_str}` template route does — plain
ISO dates (`2026-07-01`) and the aliases `today`/`yesterday`/`prev`/
`previous` — via `content.parse_date` (called inside `docs.build_journal_payload`).

- 200: `DocPayload` (see JSON shape below). `doc_type` is always
  `"journal"`, `kind` is always `"markdown"`.
- 404 `{"detail": "invalid date '<date_str>'"}`: `date_str` doesn't parse
  (includes an exhausted `prev`/`previous` alias with no earlier entry).
- 404 `{"detail": "no journal entry for '<date_str>'"}`: `date_str` parses
  to a real date but no journal file exists for it.

### `GET /api/todo`

The `journal/todos.md` doc as a `DocPayload`, via `build_doc_payload`
directly (not a bespoke builder).

- 200: `DocPayload`, `doc_type == "journal"`, `kind == "markdown"`.
- 404 `{"detail": "todo file not found"}`: no `journal/todos.md` on disk.

**Divergence from the legacy `/todo` template route, flagged:** the legacy
route calls `render_markdown(text, title="TODO", add_toc=False)` — no TOC.
`build_doc_payload` (the only builder available, out of this subtask's
boundary to change) always renders with `add_toc` at its default (`True`).
So **`DocPayload.toc` is populated for `/api/todo`**, where the legacy page
suppresses it. The SPA can simply choose not to render `toc` for the todo
view; nothing else differs (title itself isn't part of `DocPayload` at
all, so the `"TODO"` vs. `path.stem` title-override distinction is moot).

### `GET /api/doc/{path:path}`

Any document by home-relative path, as a `DocPayload` — the general-purpose
"fetch this file" endpoint (kind dispatch — markdown/text/image/drawio/
binary — happens inside `build_doc_payload`, see T23.1 handover for the
exact table).

- 200: `DocPayload`.
- 404 `{"detail": "no such document: '<path>'"}`: path traverses outside
  `home` (rejected by `content.safe_resolve`) or doesn't resolve to an
  existing file.
- **Secret docs**: if `DocPayload.is_secret` is `True` and the request isn't
  from localhost, `html`/`toc`/`text`/`language`/`raw_url` are all `None`
  (blanked by `build_doc_payload` itself). The route additionally asserts
  this contract at the route level (defense in depth, not just trusting the
  builder) — an `AssertionError` here would be a 500, i.e. a loud failure
  rather than a silent leak if the builder's contract ever regressed. On
  localhost, secret docs render normally (`is_secret=True` but content
  populated) — same network-trust-gate semantics as every other secret-file
  check in this codebase.

### `GET /api/dir` and `GET /api/dir/{path:path}`

Directory listing as a `DirPayload`. `GET /api/dir` is the home root;
`GET /api/dir/{path}` is any subdirectory.

- 200: `DirPayload`.
- 404 `{"detail": "no such directory: '<path>'"}`: path traverses outside
  `home`, doesn't exist, or isn't a directory. (Root `/api/dir` can't 404 —
  `home` always exists once the app has booted.)

Unlike `build_dir_payload` alone (which has no traversal guard — it derives
a path by string-splitting `dirs` and joining under `home`), **this route
pre-validates the path via `content.safe_resolve` before calling the
builder**, so `/api/dir/../../etc` 404s instead of resolving outside `home`.

### `GET /api/meta`

Minimal metadata for the SPA shell/header. Deliberately tiny — not a
superset of `PluginConfig` or `Settings`.

```json
{
  "today": "2026-07-14",
  "home": "smoke_home",
  "version": "0.1.0"
}
```

- `today` — ISO date (`YYYY-MM-DD`), server wall-clock `date.today()`.
- `home` — **display name only** (`home.name`, the last path component),
  never the absolute filesystem path (that's a server-local implementation
  detail with no SPA use, and mildly sensitive to expose).
- `version` — the `awiwi` package version (`pyproject.toml`'s
  `[project].version`, resolved via `importlib.metadata.version("awiwi")`),
  or `"0.0.0"` if unresolvable (e.g. an uninstalled checkout — shouldn't
  happen in practice, but never a 500).

Response model: `MetaPayload` — defined locally in `routers/api.py`, **not**
in `schemas.py` (it's a route-local shape with no builder, unlike
`DocPayload`/`DirPayload`/`SearchHit`).

Always 200 (no error path).

### `GET /api/raw/{path:path}`

Raw file bytes — the only non-JSON `/api/*` response, and the endpoint
`DocPayload.raw_url` (for `kind in ("image", "binary")`) points at.

Query params:
- `?download=1` (or any FastAPI-truthy bool value: `true`, `1`, `yes`, ...)
  → adds `Content-Disposition: attachment; filename="<name>"`. Omitted or
  falsy → served inline (no disposition header), same bytes.

Headers:
- `ETag: "<mtime_ns>-<size>"` — always set, a strong-looking but
  non-standard composite string (not a hash of content — cheap to compute,
  changes whenever the file's mtime or size changes). Quoted per HTTP
  convention.
- `If-None-Match` — if the request's header **exactly** string-matches the
  computed ETag, responds `304` with an empty body and the same `ETag`
  header repeated, no other headers. (Simple exact-string comparison, not a
  full RFC 7232 multi-value/weak-comparison implementation — one value in,
  one value out, which is all a same-origin SPA doing simple caching needs.)
- Content-Type: `mimetypes.guess_type` on the resolved path, falling back
  to `application/octet-stream` if unrecognized.

Status codes:
- 200: bytes, as above.
- 304: conditional hit, empty body.
- 404 `{"detail": "no such file: '<path>'"}`: traversal rejected or file
  missing.
- **403** `{"detail": "secret file, localhost only"}`: the resolved file's
  stem matches the secret/credential regex (`\b(secret|credential)s?\b$`,
  same pattern as `docs.py`/`pages.py`) **and** the request isn't from
  localhost. This check is **independently re-derived** in this route from
  the resolved path on disk — it does not trust or consume any `is_secret`
  field from a `DocPayload` (a `DocPayload`'s `raw_url` string is not itself
  a security control; whichever caller eventually GETs that URL still has
  to pass this gate). On localhost, secret files serve normally (200).

### `PATCH /api/checkbox`

**New** relpath-addressed protocol — deliberately *not* the same shape as
the legacy `PATCH /checkbox` (which stays exactly as-is in
`routers/actions.py`, addressed by rendered-page-endpoint string like
`/journal/2026-07-01`, mapped back to a file via
`actions.py:_file_for_endpoint`). This new endpoint is addressed directly by
`DocPayload.watch_path` (a home-relative POSIX relpath) instead, skipping
that endpoint→file indirection entirely — the reason a future `DocPayload`
consumer (this SPA) can PATCH a checkbox without knowing what URL rendered
the page it came from.

The underlying MD5 line-hash protocol (`awiwi.checkbox.hash_line` /
`toggle_checkbox`) is **completely unchanged** — only field *names* differ
from the legacy body shape.

Request body:

```json
{
  "path": "journal/todos.md",
  "line_no": 2,
  "line_hash": "3f9a1c...",
  "checked": true
}
```

- `path` — home-relative POSIX relpath (a `DocPayload.watch_path` value).
- `line_no` — 0-indexed line number (same numbering `toggle_checkbox`
  and the rendered `data-line-nr` HTML attribute already use).
- `line_hash` — MD5 hex digest from `hash_line`, as already embedded in
  rendered HTML's `data-hash` attribute (this part is byte-identical to the
  legacy protocol's `hash` field, just renamed).
- `checked` — target state (`true` → `[x]`, `false` → `[ ]`).

Maps onto `toggle_checkbox(path, line_nr, check, expected_hash)` as:
`line_no -> line_nr`, `checked -> check`, `line_hash -> expected_hash`.

Success response, 200:

```json
{
  "success": true,
  "line_hash": "3f9a1c...",
  "mtime_ns": 1752480000123456789
}
```

- `line_hash` — **echoes the request's `line_hash` back, unchanged.**
  Toggling a checkbox never changes this hash: `hash_line` strips the
  `[ ]`/`[x]` box *before* hashing, precisely so a client can chain further
  toggles without re-fetching the page. This is decided/documented behavior,
  not an oversight — a client that reads `line_hash` off this response and
  feeds it into the next PATCH doesn't need to special-case "the hash is
  the same as before."
- `mtime_ns` — the file's `Path.stat().st_mtime_ns` immediately after the
  toggle (so a client tracking a `DocPayload.mtime_ns` can update it without
  a round-trip re-fetch).

Error responses:
- 404 `{"detail": "no such file: '<path>'"}`: `path` traverses outside
  `home` (rejected by `safe_resolve`).
- 404 `{"detail": "<message>"}`: propagated from `checkbox.LineNotFoundError`
  (line at/past EOF) or a builtin `FileNotFoundError` (file doesn't exist).
- **409** `{"detail": "<message>"}`: propagated from
  `checkbox.HashMismatchError` (stale client — the file changed since the
  hash was computed), `checkbox.AlreadyInStateError` (checkbox already in
  the requested state), or `checkbox.NotACheckboxLineError` (target line
  isn't a `* [ ]`/`* [x]` item at all).

The legacy `PATCH /checkbox` action route (`routers/actions.py`) is
untouched and keeps working exactly as before, on its own body shape.

### `GET /api/search`

Ripgrep-backed content search. Query params:

- `q` (required, non-empty) — the search pattern.
- `mode` — `"fixed"` (default) or `"regex"`.
- `scope` — optional comma-separated list drawn from `journal`, `assets`,
  `recipes` (e.g. `?scope=journal,recipes`). Omitted → search all three
  (no restriction beyond the base `!awiwi*` exclude every search already
  has).

Status codes:
- 200: `list[SearchHit]` (see shape below), sorted via the existing
  `search.sort_hits` (todo, then journal, then asset, then recipe; lexical
  by name within a type). Empty list if there were no matches (**not** an
  error — ripgrep exit code 1 means "ran fine, no matches").
- **422** (FastAPI's standard validation-error shape, `{"detail": [...]}`,
  not the plain `{"detail": "<str>"}` used elsewhere in this router):
  - `q` missing or empty (`Query(min_length=1)` on a required param —
    **decided over a plain empty-list response**: an empty/missing query
    is a client mistake worth flagging distinctly from "your search legitimately
    found nothing").
  - `mode` is neither `"fixed"` nor `"regex"` (`Literal` type validation).
  - `scope` contains a token that isn't one of `journal`/`assets`/`recipes`
    — this one *is* the plain `{"detail": "invalid scope(s) [...], expected
    any of ('journal', 'assets', 'recipes')"}` shape (raised manually, not
    FastAPI's automatic validation), since it can't be expressed as a
    single-value `Literal` on a comma-joined string.
- **400** `{"detail": "<rg's stderr, or 'invalid search pattern'>"}`: ripgrep
  exited with code 2 (a genuine pattern/usage error — e.g. malformed regex
  like `(unclosed` under `mode=regex`) rather than exit 0 (matches) or exit 1
  (no matches). This is the "clean 400, not a 500" the design contract asked
  for.
- 500 `{"detail": "search failed: <exc>"}`: `rg` itself couldn't be spawned
  (e.g. not installed) — an `OSError` from `subprocess.run`, not a
  ripgrep-reported error.

`SCOPE_DIRS` and the glob-translation (`{scope}/**`) live in `search.py`
alongside the extended `build_rg_args`.

#### `search.build_rg_args` extension (the only `search.py` change)

```python
def build_rg_args(
    pattern: str, *, fixed: bool = False, scopes: Sequence[str] | None = None
) -> list[str]: ...
```

- `fixed=True` appends `-F` (ripgrep: treat `pattern` as a literal string,
  not a regex).
- `scopes` — an iterable of top-level dir names — appends one `-g
  "{scope}/**"` pair per scope, before the trailing `pattern` argument.
- **`fixed` defaults to `False`, deliberately** — this reproduces the
  *exact* argv the legacy `POST /search/content` action route has always
  gotten from its existing positional-only call, `build_rg_args(pattern)`
  (no `-F`, i.e. always-regex mode, unchanged since T14). The `/api/search`
  route itself defaults its own `mode` query param to `"fixed"` and passes
  `fixed=True` explicitly when constructing its own call — so the *new*
  route's stated default ("`-F` when fixed (default)" per the design brief)
  is honored without changing the *function's* default and silently
  altering the legacy route's behavior.
- `parse_search_output`/`sort_hits` are completely untouched.

### `SearchHit` JSON shape

```json
{
  "target": "/journal/2026-07-01",
  "name": "2026-07-01",
  "line": 3,
  "col": 5,
  "type": "journal",
  "text": "buy milk today"
}
```

Field-for-field copy of `search.SearchHit` (the existing dataclass);
`type` is one of `"todo"`/`"journal"`/`"asset"`/`"recipe"`. `target` is a
legacy-template-route-style path (e.g. `/journal/2026-07-01`,
`/assets/2026-07-01/x.txt`) — **not** a `watch_path`-style relpath and
**not** an `/api/doc/...` URL. Flagging for the SPA: if the search results
UI wants to navigate to a hit via `/api/doc/...`, it'll need its own
relpath-from-target mapping (not provided here — out of this subtask's
scope, `SearchHit` is a faithful copy of the existing dataclass, not a
redesign).

## `DocPayload` JSON shape (reference — full field docs in T23.1 handover)

```json
{
  "kind": "markdown",
  "doc_type": "journal",
  "html": "<p>...</p>",
  "toc": "<div class=\"toc\">...</div>",
  "text": null,
  "language": null,
  "raw_url": null,
  "watch_path": "journal/2026/07/2026-07-01.md",
  "breadcrumbs": [{"name": "journal", "target": "/dir/journal"}, ...],
  "journal_date": null,
  "nav": {"prev": "2026-06-30", "next": "2026-07-02"},
  "mtime_ns": 1752480000123456789,
  "is_secret": false
}
```

`kind`: `"markdown" | "text" | "image" | "drawio" | "binary"`.
`doc_type`: `"journal" | "asset" | "recipe" | "other"`.
See T23.1 handover (`handovers/server-rewrite/T23.1-payload-builders.md`)
for the exact per-field derivation/kind-dispatch table and the secret-gate
blanking rules — unchanged by this subtask.

## Deviations from the design contract, assessed

1. **`build_rg_args`'s `fixed` parameter defaults to `False`, not `True`.**
   The design contract's literal wording ("`-F` when fixed (default)")
   reads naturally as "the function should default to fixed mode." But the
   *function* is also called, unchanged, by the legacy `POST
   /search/content` action route (`build_rg_args(pattern)`, positional,
   no keywords) — flipping the function's own default would silently
   change that existing route's behavior (always-`-F`), which the task's
   hard boundary ("old template routes stay untouched and byte-identical")
   forbids. Resolved by keeping the function's default `False` (verified
   unchanged via `test_default_unchanged_for_legacy_callers`, byte-identical
   to the pre-existing `test_matches_legacy_flags`) and having the `/api/search`
   *route* supply its own `fixed=True` default via its `mode` query param's
   default value (`"fixed"`). End-to-end behavior at `/api/search` matches
   the contract; the shared function's default doesn't regress its only
   other caller.
2. **`/api/todo` doesn't suppress `toc`** the way the legacy `/todo`
   template route does (`add_toc=False`) — `build_doc_payload` (the only
   builder in the S23.1 boundary, not touchable here) always renders with
   `add_toc` at its default (`True`). See the `/api/todo` section above.
   Flagging for whoever eventually does the "moved not rewritten" real
   extraction (T23.1 handover's flagged future cleanup) — that'd be the
   natural place to give `build_doc_payload` an `add_toc` override too.
3. **Empty/missing `q` on `/api/search` → 422, not an empty list** — chosen
   because a query the client forgot to fill in is a different situation
   from "your search legitimately found nothing," and FastAPI's `Query(min_length=1)`
   gives this for free with the standard validation-error shape.
4. **Non-`GET` requests to unmatched `/api/*` paths aren't guaranteed a
   JSON 404** — only `GET /api/{rest:path}` exists as the catch-all. See
   Route-ordering section above; not exercised by any test, flagged as a
   possible future tightening rather than fixed now (all real endpoints
   below `/api` are correctly method-guarded already; this only affects
   genuinely nonexistent path+method combinations).
5. **`ETag` is a composite `"{mtime_ns}-{size}"` string, not a content
   hash** — cheap to compute (no file read needed to answer a
   conditional-GET check), same tradeoff class as most static file servers.
   Comparison is exact-string only (no weak-validator `W/` handling, no
   multi-value `If-None-Match` list parsing) — suffices for a same-origin
   SPA's own cache-revalidation use, not a general HTTP cache proxy.
