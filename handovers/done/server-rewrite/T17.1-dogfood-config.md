# server-rewrite / T17.1 — dogfood config fix

**Responsibility:** fix the lifespan crash on the user's first live `:Awiwi serve`
(S17.3 dogfood): `PluginConfig` rejected the real `config.json`.
**Boundary:** `lua/awiwi/init.lua` (get_markers rebind), `tests/server_spec.lua`,
`server/src/awiwi/config.py`, `server/tests/test_config.py`, plus ledger/docs.

## What downstream needs from me

- **B14 (plugin side, root cause):** T10's façade wiring rebound
  `server.config.get_markers = markers.get_markers`, silently dropping the legacy
  shim's `{ join = false }` — so `_write_json_config` wrote pipe-joined marker
  *strings* into config.json instead of the *arrays* the vimscript original (and
  T13's model) used. Fixed: the rebind now wraps with `join = false`. New spec
  reloads the façade (`package.loaded["awiwi"] = nil`) because both `reset()` and
  the older config.json spec stub `get_markers` with list-returning fakes — that's
  how the suite masked the regression.
- **Server side, two hardenings in `config.py`:**
  - `screensaver: str | bool = False` — the plugin writes `vim.g.awiwi_screensaver`
    verbatim, which is a screensaver *name* string (e.g. `"cinnamon"`) in real
    configs. T13 modeled it as bool on a wrong guess.
  - `PluginConfig.load` now catches `ValidationError`/`OSError` → logs a warning and
    returns defaults. "Parsed permissively" now includes *unparseable*, not just
    missing/extra-keys: a bad config.json can never prevent boot. The plugin
    rewrites config.json on every `start_server`, so a healthy launcher self-heals
    the file.
- Nothing consumes `screensaver`/markers server-side yet; the model is protocol
  description only.
