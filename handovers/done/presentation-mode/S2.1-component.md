# presentation-mode / S2.1 PresentationMode.svelte

**Responsibility:** Fullscreen slide-deck overlay component. Given a way to reach the
live, enhanced `.markdown-body` DOM, it splits it into slides (via S1.1's
`slides.ts`), enters real Fullscreen, and lets the user step through slides with
keyboard, mouse, or on-screen arrows.

**Boundary:** `server/frontend/src/lib/components/PresentationMode.svelte` (new, 179
lines). `server/frontend/src/app.css` (`/* presentation mode */` block appended
after the lightbox block, ~30 rules). Nothing else touched.

## Component contract (for S2.2, wiring into DocPage.svelte)

```svelte
<script lang="ts">
  import PresentationMode from "./PresentationMode.svelte";
  let pm: PresentationMode | undefined;
</script>

<PresentationMode bind:this={pm} getRoot={() => articleEl?.querySelector(".markdown-body") ?? null} />
```

- Prop: `getRoot: () => Element | null`. Called once, inside `open()`. Return the
  live `.markdown-body` element (the same one `MarkdownView.svelte` renders into) —
  or `null` if there's nothing to present, in which case `open()` is a no-op.
- Exported methods via `bind:this`: `open()` and `close()`. No other public surface.
- Import path used inside the component: `../presentation/slides` — **this repo
  does not use a `$lib` alias anywhere** (checked: zero hits across `src/`), all
  cross-module imports are relative. Followed that convention; don't introduce
  `$lib` in S2.2 either.
- `open()` snapshots slides from `getRoot()` at call time — if the doc changes
  underneath (live-reload via ws), the deck does NOT auto-refresh. Not in scope
  per the plan; flag if S2.2/S3 wants live-refresh behavior.

## Behavior implemented (per the approved plan)

- Fullscreen: `open()` sets `visible = true`; a **separate** `$effect` (keyed
  only on `visible`/`overlayEl`) calls `overlayEl.requestFullscreen().catch(() =>
  {})` once the overlay div has actually mounted. **Gotcha found and fixed
  during implementation:** calling `requestFullscreen()` directly inside `open()`
  (as the plan's pseudocode literally read) is a bug — `overlayEl` (bound via
  `bind:this` inside `{#if visible}`) is not yet assigned at that point in the
  synchronous call, since the block hasn't rendered. It must be deferred to an
  effect that runs after the DOM update. Kept it in its own effect (rather than
  folding into the listener effect) so a bare `overlayEl` rebind doesn't churn
  the keydown/mousemove/fullscreenchange listeners.
- `fullscreenchange` listener closes the overlay when `document.fullscreenElement`
  clears (covers native Esc). Plain `Escape` keydown is also handled as a
  fallback for when fullscreen was rejected. `close()` calls
  `document.exitFullscreen()` if still active.
- Transition: `{#key index}` + `transition:fly`, direction-aware via `dir` (±1,
  set in `next()`/`prev()`). Duration computed in JS
  (`window.matchMedia("(prefers-reduced-motion: reduce)")`) since Svelte
  transition durations are plain numbers and don't see the app.css `--dur-*`
  zeroing.
- Navigation: ArrowLeft/ArrowRight/Space (Space does `preventDefault`), overlay
  left-click advances unless the click `.closest()`-matches
  `a, button, input, summary, textarea, select` (in-slide interactive elements
  pass through untouched). `step()`/`arrowOpacity()` from `slides.ts` drive
  clamping and arrow dimming — no wraparound.
- Controls: `mousemove` sets `controlsVisible = e.clientY >= innerHeight * 0.8`
  (deliberately not CSS `:hover`, which would eat the bottom-fifth
  click-to-advance region). Fixed bottom-right column: prev/next buttons
  (`.btn.btn-icon`, opacity from `arrowOpacity`, `disabled` at edges), and below
  them a centered `.btn.btn-ghost` labeled `" Normal mode"` → `close()`. The
  `.pm-controls` wrapper stops click propagation so control clicks never also
  trigger click-to-advance.
- All listeners (`keydown`, `mousemove`, `fullscreenchange`) attached only in the
  `$effect` while `visible`, torn down in its cleanup — mirrors
  `Lightbox.svelte`'s pattern exactly.

## Deviation flagged: the exit-button label glyph

The plan specified the secondary button text as `" Normal mode"` with "leading
char verbatim, assumed nerd-font glyph". As received, the only leading
character present in the plan text was a plain space (U+0020) — no nerd-font
private-use-area codepoint was visible/recoverable in the instruction. I
implemented it literally as given: a single leading space before "Normal mode"
(HTML whitespace collapses to one space on render regardless of source
indentation, so this is visually a plain space, not an icon). **If the
intended design has an actual nerd-font glyph (e.g. an "exit fullscreen" /
"windowed" icon) before the text, S2.2 or a follow-up should supply the
specific codepoint** — I had no source to recover it from and did not want to
guess a plausible-but-wrong PUA codepoint into shipped code. This app's font
stack (`--font-ui`/`--font-body`/`--font-mono`) does not declare a nerd font;
a glyph would only render if the user's OS has one installed as a system-level
fallback font (this may be intentional per "assumed nerd-font glyph").

## CSS added (`app.css`, after the lightbox block)

- `.pm-overlay` — `position: fixed; inset: 0`, `background: var(--bg-canvas)`,
  flex column, `z-index: 200` (above `.lightbox-overlay`'s `z-index: 100` —
  the two are never shown together, but this future-proofs stacking if they
  ever are).
- `.pm-stage` — `position: relative`, `flex: 1`, `overflow: hidden` (the
  positioning context for absolutely-positioned `.pm-slide`s during the fly
  transition).
- `.pm-slide` — `position: absolute; inset: 0`, `overflow-y: auto` (scrollable
  if a slide is tall), `max-width: min(80vw, 900px)`, centered, padded.
- `.pm-slide .copy-btn-float { display: none; }` — hides the floating
  code-block copy buttons that `enhance/copyButtons.ts` bakes into the live
  `.markdown-body`'s `outerHTML` (and thus get cloned verbatim into slide
  HTML) — slides are read-only/inert, per the plan.
- `.pm-controls` — `position: fixed`, bottom-right, `z-index: 201`; `.pm-arrows`
  (flex row, bumped `.btn` font-size for the ‹ › glyphs); `.pm-exit` (font-size
  only, relies on `.btn.btn-ghost` for everything else).
- All values use existing Noir-Deco custom properties (`--bg-canvas`,
  `--space-*`, `--text-lg`, `--text-sm`) — no new tokens introduced.

## Gotchas for S2.2 (mounting in DocPage.svelte)

1. `getRoot` must resolve to the *same* `.markdown-body` element
   `MarkdownView.svelte` owns (`bind:this={container}` in that component) —
   likely via `articleEl.querySelector(".markdown-body")` from DocPage's
   existing `articleEl` binding, or by exposing the container ref through
   `MarkdownView` if a cleaner handle is wanted.
2. `PresentationMode` renders nothing until `open()` is called (no `{#if}`
   wrapper needed around the `<PresentationMode>` tag itself — mount it
   unconditionally, call `.open()` from wherever the "present" trigger lives).
3. Fullscreen requires a user gesture — call `.open()` directly from a click
   handler (button/menu item), not from an effect or timeout, or
   `requestFullscreen()` will silently reject (caught, so it just degrades to
   the CSS `position: fixed` overlay per the plan's fallback design).
4. z-index 200/201 sit above the lightbox (100) and the app header — verify no
   other overlay in DocPage's tree needs to sit above presentation mode.
