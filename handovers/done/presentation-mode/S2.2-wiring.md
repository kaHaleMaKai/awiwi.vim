# presentation-mode / S2.2 wiring: PresentationMode.svelte into DocPage.svelte

**Responsibility:** Mount S2.1's `PresentationMode` inside `DocPage.svelte`'s markdown
branch and give the user a trigger to open it: a "Present" button in the rail, shown only
for `doc_type === "asset"` docs that have a TOC.

**Boundary:** `server/frontend/src/lib/components/DocPage.svelte` only (+15/-0 lines).
Nothing else touched.

## What was wired, and where

All changes in `server/frontend/src/lib/components/DocPage.svelte`:

- Line 15: `import PresentationMode from "./PresentationMode.svelte";`
- Line 63: `let presenter: PresentationMode | undefined = $state();` — next to the existing
  `articleEl`/`tocEl` bindings.
- Lines 118–124: inside the existing `{#if doc.toc}` `aside.rail` block (which already
  contains the "On this page" TOC `.rail-section`), a **second** `.rail-section`, gated
  `{#if doc.doc_type === "asset"}`:
  ```svelte
  <div class="rail-section">
    <button class="btn btn-ghost" type="button" onclick={() => presenter?.open()}>
       Present
    </button>
  </div>
  ```
  (Leading space before "Present" is verbatim, per S2.1's contract — placeholder for a
  nerd-font glyph, same unresolved-glyph situation as PresentationMode's own
  " Normal mode" exit button.)
- Lines 128–131: `<PresentationMode>` mounted unconditionally (per S2.1's contract — no
  `{#if}` wrapper needed) just after the closing `{/if}` of the TOC-gated aside, still
  inside `.layout-with-rail`:
  ```svelte
  <PresentationMode
    bind:this={presenter}
    getRoot={() => articleEl?.querySelector(".markdown-body") ?? null}
  />
  ```

`articleEl` (bound to the `<article class="stack">` wrapper, line ~92, pre-existing) was
reused as-is — no new element binding was needed. `MarkdownView.svelte` renders
`<div class="markdown-body" bind:this={container}>` inside that article, so
`articleEl.querySelector(".markdown-body")` resolves to the live enhanced markdown root
S2.1 expects.

**No TOC → no rail → no button**, confirmed: the Present button lives strictly inside the
pre-existing `{#if doc.toc}` gate, so docs without a TOC never render it, per plan.
`doc.doc_type === "asset"` is a valid `DocType` literal (`server/frontend/src/lib/api.ts:6`
— `"journal" | "asset" | "recipe" | "other"`), so the gate type-checks cleanly.

## Manual smoke checklist (for a human to run)

1. Open an asset doc with multiple top-level `# H1` headings and a TOC (i.e.
   `doc_type === "asset"`, non-empty `doc.toc`).
2. Confirm the rail shows two sections: "On this page" (TOC) and a new one with a
   " Present" button below it.
3. Click "Present" → overlay goes fullscreen, slide 0 shown.
4. Press `Right` / `Space` / click the slide body (not on a link/button/input) → advances
   to slide 1, etc.
5. Press `Left` → goes back a slide.
6. At slide 0, confirm `Left`/prev-arrow does nothing (clamped, no wraparound); at the
   last slide, confirm `Right`/next-arrow does nothing.
7. Press `Esc` → exits fullscreen and closes presentation mode, back to normal DocPage.
8. Move mouse to bottom fifth of the screen while presenting → prev/next arrows and
   " Normal mode" button appear; move mouse away from bottom fifth → controls hide.
9. Click " Normal mode" button while presenting → exits fullscreen, closes overlay, back
   to normal DocPage view.
10. Confirm a recipe or journal doc (non-asset `doc_type`) never shows the "Present"
    button, even with a TOC.
11. Confirm a doc with no TOC shows no rail at all (so no Present button either), even if
    `doc_type === "asset"`.

(Steps 3–9 exercise S2.1's own internals — already covered by that unit's implementation
— this checklist is about confirming the *wiring*: button visible in the right place,
`presenter.open()` reachable, `getRoot` resolves the right element.)
