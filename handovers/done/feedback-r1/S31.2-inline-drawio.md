# S31.2 — inline-render drawio diagrams linked from journal bodies

## Stakeholder feedback + interpretation

Verbatim (garbled): "if in a journal there is a link to a drawio diagram, and show the
link as a sub text".

**Orchestrator interpretation (adopted):** when a rendered doc body contains a plain
link `<a href="….drawio">label</a>`, replace it with the inline-rendered diagram
(self-hosted viewer, same machinery as the drawio asset page) and keep the original
link as a subtext caption beneath the diagram — still navigable.

## Design

### New enhance pass — `server/frontend/src/lib/enhance/drawio.ts`

- Selects `a[href$=".drawio"]` in the enhanced container; skips anchors already
  marked (`data-drawio-inline`) or already inside a rendered `.drawio-inline` figure.
- Href resolution mirrors `media.ts` (`resolveDrawioPath`, exported + unit-tested):
  `assets/…` is already home-relative; any other relative href resolves against the
  doc's directory (`watchPath`); absolute/scheme/root hrefs are left untouched.
- Fetches the diagram XML via the existing `getDoc(relpath)` helper (the `/api/doc`
  payload carries the XML in `doc.text` for `kind === "drawio"` — same source the
  asset page uses), then awaits the shared viewer script and swaps the anchor for:

  ```html
  <figure class="drawio-inline" data-drawio-inline="done">
    <div class="mxgraph" data-mxgraph='{"xml":…, "lightbox":false}'></div>
    <figcaption><a href="…">original label</a></figcaption>
  </figure>
  ```

  followed by `window.GraphViewer.processElements()`.
- **Security decision preserved:** `lightbox: false`, no "Open in draw.io" — nothing
  ever loads from app.diagrams.net (same rationale as DrawioView / vendor README).
- **Graceful degradation:** resolve/fetch/script failure leaves the original `<a>`
  fully intact (no partial DOM mutation on the failure path); the anchor is marked
  `data-drawio-inline="error"` so re-enhance doesn't retry-loop it.
- **Idempotency:** dataset flag set the moment processing starts + the
  `.closest(".drawio-inline")` exclusion; a second `enhance()` over the same DOM does
  not re-fetch or double-wrap. WS re-renders that replace the `{@html}` content get
  fresh anchors and correctly re-render.
- Returns a no-op cleanup (the pass mutates the DOM once, attaches no listeners),
  matching the pipeline's cleanup contract.

### Singleton loader extraction — `server/frontend/src/lib/drawioViewer.ts`

The lazy singleton `<script>` injection for `vendor/drawio/viewer-static.min.js`
(formerly a `<script module>` block in `DrawioView.svelte`, incl. the
`window.GraphViewer` global declaration) moved verbatim to `src/lib/drawioViewer.ts`
as `loadDrawioViewer()`. `DrawioView.svelte` now imports it; its behavior is
otherwise identical. One implementation, one script tag app-wide, shared by the
component and the enhance pass.

### Wiring + styles

- `src/lib/enhance/index.ts`: `cleanups.push(enhanceDrawio(container, opts.watchPath))`
  directly after `rewriteMedia` (needs the same relative-href world the media pass
  operates in; order before redaction is immaterial — passes are independent).
- `src/app.css`: `.markdown-body figure.drawio-inline` (sunken canvas consistent with
  the mermaid box: `--bg-sunken`, `--border-strong`, `--radius-md`, `overflow-x:
  auto`) + centered muted `--text-sm` figcaption. Existing tokens only, both themes
  covered automatically.
