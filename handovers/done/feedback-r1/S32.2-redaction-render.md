# S32.2 — redaction embed reveals RENDERED markdown

**Status: DONE.** All gates green (263 passed / ruff clean / basedpyright 0 errors 0 warnings).

## ⚠ CONTRACT CHANGE (for kb-curator)

The S23.4 byte-round-trip contract for **heading-section** redaction embeds is
replaced. `docs/architecture.md`'s redaction paragraph must be updated (and an
ADR appended if the curator judges it decision-worthy — this supersedes the
"hidden content never passes through python-markdown, revealed value reads back
character-for-character" rule *for sections only*; ADR D23 localhost-only
posture unchanged).

### Old contract (S23.4)

`<div class="redacted">` contained the hidden section's **raw lines,
HTML-escaped**, stashed behind an inert token and substituted after
`md.convert` — no markdown rendering, no pre-filters, byte-fidelity on reveal.

### New contract (S32.2)

`<div class="redacted">` contains the **rendered HTML of the hidden section**:

```html
<h3 id="redacted"><em>…redacted…</em></h3>          <!-- obscured placeholder, unchanged -->
<div class="redacted"><h3 id="at-media-ai-tech-hub">AT Media: AI + tech hub</h3>
<p><a href="../../../assets/2026/06/23/at-media-prompt.md">AT Media prompt</a></p></div>
```

- The section's own heading, with `!!redacted` (and any trailing cause text)
  stripped, at its original h-level, rendered first inside the div.
- The body rendered through the exact same pipeline as visible content:
  per-line pre-filters (checkboxes, `@tag`/`@@mention`/`#tag` spans, ordinal
  superscripts, fence tracking) + a fresh `_new_markdown()` instance.
- Raw inline/block HTML in the hidden body now **passes through** like it does
  in visible content (previously escaped). Sanctioned: localhost-only, ADR D23.
- Token mechanism kept: the pre-rendered fragment is stashed behind the inert
  `awiwiredacted<uuid>n<i>` token and substituted after the **outer**
  `md.convert`, so it never travels through the outer markdown pass.

## Design decisions

1. **Nested pre-filters with real line numbers** — `_flush_hidden` re-runs
   `_filter_body(hidden_lines, offset=hidden_start, embed_redacted=False)`
   where `hidden_start` is the redacted heading's original 0-indexed file line
   and the buffer is the contiguous section (stripped heading + verbatim body).
   Checkbox `data-line-nr`/`data-hash` inside the embed therefore satisfy the
   PATCH /checkbox contract exactly (pinned by
   `test_section_embed_applies_prefilters_with_original_line_numbers`).
   No limitation shipped.
2. **Nested `!!redacted` markers: stripped, not embedded** — the nested pass
   runs with `embed_redacted=False`, i.e. legacy stripping. Doubly-redacted
   content stays hidden even after reveal, and recursion is structurally
   impossible (the nested pass plants no tokens, renders no divs).
3. **Inline form unchanged** — `<span class="redacted">` still carries the
   HTML-escaped raw value. Rationale: inline redactions are typically secrets
   (API keys, passwords) where character-for-character read-back matters more
   than rendering (`pass*word*123` must not lose its asterisks). The feedback
   case is the section form. Documented in `_filter_body`'s docstring.
4. **Obscured presentation unchanged** — placeholder heading
   `<marker> _…redacted…_` + toggleable `.redacted` element, one div per
   section; frontend untouched.
5. **Non-embed mode (allow_remote) byte-identical** — pinned by the untouched
   `test_default_is_byte_identical_to_stripped_behavior` and the
   TestRedactionEmbedGating strip tests.
6. Known minor edge (accepted, untested): footnote ids inside a hidden section
   come from a separate Markdown instance and could in theory collide with the
   outer document's footnote ids; same class of collision already possible for
   TocExtension heading ids. Not observed in real notes.
