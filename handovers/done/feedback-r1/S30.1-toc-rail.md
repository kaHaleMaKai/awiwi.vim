# S30.1 — TOC rail layout fixes

Stakeholder feedback: rail gets pushed off-screen on wide-body articles; make
it stick to the right edge; make it collapsible on click; collapse it by
default below 700px viewport width.

## Root cause (confirmed)

`.layout-with-rail { grid-template-columns: 1fr 310px; }` — the `1fr` article
track has an implicit `min-width: auto`, so non-wrapping wide content
(`<pre>`, `<table>`) inside the article expands the track past its fraction,
which shoves the fixed 310px rail column off the right edge of the viewport.
Confirmed no `.markdown-body pre`/`.markdown-body table` overflow-x handling
existed for the pre-enhancement state (Shiki's injected `.shiki` rule and
`.code-block pre`, used by `TextFileView`, both already had `overflow-x:
auto`, but the raw `<pre>` markdown-body emits before `copyButtons.ts`/
`shiki.ts` enhance it did not).

## Changes

### `server/frontend/src/app.css`

1. **Overflow fix**: `.layout-with-rail` track changed to
   `minmax(0, 1fr) 310px` — wide content now scrolls inside the article
   instead of stretching the grid track. Added the two minimal missing
   rules: `.markdown-body pre { overflow-x: auto; }` and
   `.markdown-body table { display: block; overflow-x: auto; max-width:
   100%; }` (the `display: block` on `table` is required for the
   `overflow-x` to take effect while keeping `border-collapse`/cell styling
   intact — a standard responsive-table technique).
2. **Sticks to the right edge**: no change needed beyond the overflow fix —
   the rail is already the grid's last column with `position: sticky`
   (vertical pin only); once the overflow bug stopped displacing the whole
   grid, "last column, sticky" already reads as "stuck to the right edge" of
   the layout. (The layout itself lives inside `.container-wide`, max-width
   1240px and centered, so on very wide monitors "right edge" means the
   container's edge, not the literal OS screen edge — judged good enough,
   not a literal `position: sticky; right: 0` against the viewport, since
   that would fight the centered container.)
3. **Collapsible**: new `.rail-header`, `.rail-toggle` (28x28 button),
   `.rail.is-collapsed` (hides `.deco-title` + `.toc`, leaves the toggle as
   a slim affordance), and `.layout-with-rail.rail-collapsed` (shrinks the
   grid column to `auto`, i.e. just the button's width).
4. **Breakpoints**: kept the existing `@media (max-width: 900px)` stacking
   rule (single column, rail static) unchanged in effect, just extended to
   also flatten `.rail-collapsed`'s columns. Added a `@media (max-width:
   700px)` block that intentionally carries no CSS rules of its own — it's
   there to document the default-collapse breakpoint, since the actual
   default-collapse behavior is JS-seeded state (see below), not a pure CSS
   rule. Reconciliation: 900-700px is single-column/stacked but expanded by
   default; below 700px it's single-column/stacked and collapsed by
   default; the user's toggle always overrides the default at any width.

### `server/frontend/src/lib/components/JournalPage.svelte`

- New `railCollapsed` state, seeded once at component init from
  `window.matchMedia("(max-width: 700px)").matches` (no resize listener —
  session-only default, not re-evaluated live; **no persistence**, per
  instructions — a page reload re-seeds from the media query).
- `toggleRail()` flips it; wired to a new `<button type="button"
  class="rail-toggle" aria-expanded={!railCollapsed} aria-label="...">`
  inside a new `.rail-header` wrapper around the existing `On this page`
  `.deco-title`.
- `class:rail-collapsed={railCollapsed}` on `.layout-with-rail`,
  `class:is-collapsed={railCollapsed}` on `.rail` — pure class toggles, the
  `<aside>`/`<nav class="toc">`/`tocEl` DOM stays mounted whether collapsed
  or not (only hidden via CSS), so `tocSpy.ts`'s `watchToc()` (bound once to
  `tocEl` in the existing `$effect`) is unaffected — verified by reading
  `watchToc`: it queries `toc.querySelectorAll("a[href^='#']")` once at
  setup and observes article headings via `IntersectionObserver`, neither of
  which cares whether the rail is CSS-hidden. No changes made to
  `tocSpy.ts`.

### Out of scope (per boundary), noted for awareness

`DocPage.svelte` and `TodoPage.svelte` share `.layout-with-rail`/`.rail`
markup and therefore silently inherit the overflow fix and sticky/right-edge
behavior from the CSS change (good — same bug class applies there). They do
**not** get the collapse toggle button/state, since boundary restricted
script changes to `JournalPage.svelte` only. If the collapse UX is wanted
fleet-wide, a follow-up unit should extract the toggle button + state into a
small shared piece and wire it into those two pages too.
