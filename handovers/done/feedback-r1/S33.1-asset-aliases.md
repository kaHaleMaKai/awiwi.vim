# S33.1 — asset URL alias mappings

## Stakeholder feedback (verbatim)

> URL path mapping needs improvements for assets: we need to add alias mappings to match
> the format expected in the journals and other assets
> - `/assets/YYYY-MM-DD/filename.ext` (proper path)
> - `/assets/YYYY/MM/DD/YYYY-MM-DD/filename.ext` (new alias)
> - `../assets/YYYY/MM/DD/YYYY-MM-DD/filename.ext` (new alias)
> - `../../assets/YYYY/MM/DD/YYYY-MM-DD/filename.ext` (new alias)
> - `../../../assets/YYYY/MM/DD/YYYY-MM-DD/filename.ext` (new alias)

## Design

On disk, assets live at `assets/{year}/{month}/{day}/{name}` (`docs.py`'s `journal_date`
derivation, `"-".join(path.parent.relative_to(assets_root).parts)`, confirms this). The
SPA itself already links/serves assets via this disk shape at the API level (`raw_url` /
`watch_path` in `DocPayload` are `assets/YYYY/MM/DD/name.ext` — see
`tests/test_api.py:200,203` and `tests/test_acceptance.py`), so that shape had to keep
working unchanged.

The `../`-prefixed variants in the feedback are **not** a distinct server-side shape:
markdown links are emitted verbatim (no href rewriting in `mdrender.py`), authored
relative to the *file on disk*, but resolved by the browser against the *SPA page URL*
(different tree depth) — by the time a request reaches the server, those `../` segments
have already collapsed into one of the absolute shapes below. Nothing extra was needed
for them.

### Alias table (canonical = on-disk relpath `assets/YYYY/MM/DD/name.ext`)

| Shape | Canonicalizes to | Applied via |
|---|---|---|
| `assets/YYYY/MM/DD/name.ext` | itself (already disk shape) | `content.normalize_asset_path` (no-op) |
| `assets/YYYY-MM-DD/name.ext` | `assets/YYYY/MM/DD/name.ext` | `content.normalize_asset_path` |
| `assets/YYYY/MM/DD/YYYY-MM-DD/name.ext` (Y/M/D must agree with the dashed date; else passed through unchanged → 404 downstream) | `assets/YYYY/MM/DD/name.ext` | `content.normalize_asset_path` |
| any non-`assets/...` path, or a non-matching `assets/...` shape | unchanged | `content.normalize_asset_path` (no-op passthrough) |
| SPA page URL `/assets/YYYY/MM/DD/file` | 302 → `/assets/YYYY-MM-DD/file` | `redirects.asset_ymd_redirect` (pre-existing) |
| SPA page URL `/assets/YYYY/MM/DD/YYYY-MM-DD/file` | 302 → `/assets/YYYY-MM-DD/file` (redirects regardless of whether the dashed segment agrees with Y/M/D — the SPA page route only understands the canonical dashed shape) | `redirects.asset_ymd_dashed_redirect` (new) |

`../`, `../../`, `../../../`-prefixed variants: resolved by the browser before the
request is sent; they collapse into one of the absolute shapes above and need no
separate handling.

## Where it's applied

- **`server/src/awiwi/content.py`** — new pure function `normalize_asset_path(relative:
  str | Path) -> str`. Two regexes (`_ASSET_YMD_DASHED_RE`,
  `_ASSET_YMD_REDUNDANT_DASHED_RE`), longest/most-specific pattern checked first. Pure
  string manipulation, no filesystem access, no traversal guarding — callers **must**
  still run `safe_resolve` afterwards (this function only reshapes the string; it does
  not validate it stays inside `home`).
- **`server/src/awiwi/routers/api.py`** — `GET /api/doc/{path:path}` and `GET
  /api/raw/{path:path}` both call `normalize_asset_path(path)` before `safe_resolve`, so
  every alias 200s through the existing JSON/bytes response exactly as the disk-shape
  path already did (`watch_path`/`raw_url` in the `DocPayload` response are always the
  canonical disk shape, regardless of which alias the request used).
- **`server/src/awiwi/routers/redirects.py`** — new route
  `GET /assets/{year}/{month}/{day}/{_dashed}/{file}` (`asset_ymd_dashed_redirect`),
  registered alongside the pre-existing `GET /assets/{year}/{month}/{day}/{file}`
  (`asset_ymd_redirect`); different path-segment counts so there's no route-ordering
  conflict between the two. Both 302 to the same canonical `/assets/YYYY-MM-DD/file`
  page URL the SPA router understands.

## Tests added (RED first)

- `server/tests/test_content.py` — new `TestNormalizeAssetPath` (8 cases): disk shape
  passthrough, dashed→disk, redundant-dashed-segment drop, nested filename preserved for
  both mapped shapes, mismatched redundant segment passes through unchanged, non-asset
  path passthrough, non-matching asset shape passthrough, `Path` object input accepted.
  Confirmed RED via `ImportError: cannot import name 'normalize_asset_path'` before
  implementing the function.
- `server/tests/test_acceptance.py` — `test_asset_ymd_dashed_redirects_to_dashed` (302
  for the new page-URL alias) + two parametrized end-to-end tests,
  `test_api_doc_serves_all_asset_alias_shapes` and
  `test_api_raw_serves_all_asset_alias_shapes`, each covering all three shapes
  (`assets/2026/07/01/photo.png`, `assets/2026-07-01/photo.png`,
  `assets/2026/07/01/2026-07-01/photo.png`) — 200s via `/api/doc` and `/api/raw`
  respectively, asserting the response always reports the canonical
  `watch_path`/`raw_url`.

## Boundary respected

Touched only: `content.py` (new function + import), `routers/api.py` (2-line
`normalize_asset_path` wiring in `/api/doc` and `/api/raw`), `routers/redirects.py` (1 new
route), `tests/test_content.py`, `tests/test_acceptance.py`. Did not touch `mdrender.py`,
`checkbox.py`, `docs.py`, `schemas.py`, or any frontend file.

## Anything the frontend might still need

Nothing required — the frontend's own asset links/redirects were not changed and the
disk shape it already uses for `raw_url`/`watch_path` is unchanged. If the frontend (or
journal-authoring tooling) ever starts *emitting* markdown links in the
`assets/YYYY/MM/DD/YYYY-MM-DD/name.ext` or `assets/YYYY-MM-DD/name.ext` shapes itself
(rather than only receiving them from hand-authored notes), no further server change is
needed — both are now first-class aliases at both the `/api/doc` + `/api/raw` layer and
(for page-URL navigation) the redirect layer.
